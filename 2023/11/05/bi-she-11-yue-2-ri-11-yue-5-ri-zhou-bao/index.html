Research on Improved General Property Graphs Query Method Based on TinkerPop
背景

目前主流的知识图谱数据模型是 RDF 图和属性图；笔者提出了一种基于通用图框架Tinkerpop的属性图查询方法，该方法是使用RDF 图的查询语言 SPARQL 来查询属性图；并且实现了 ASK 查询和路径查询功能。

主要工作

Tinkerpop支持Gremlin，SPARQL-Gremlin模块已有；笔者主要实现query-rewrite，query-rewrite主要包括 ASK 查询和路径查询；

  • ASK 查询：根据查询条件查询数据是否存在；笔者通过对 SPARQL 进行关键字抽取，根据不同的查询类型执行相应的查询策略；根据我的理解就是，SPARQL 支持 ASK 查询，但是Jena框架无法解析 ASK 查询，不支持，所以笔者对 SPARQL 进行关键字抽取，判断是否为 ASK 查询，如果是的话，在确保原始查询语义的前提下，重写SPARQL 语句为可被Jena解析的形式，执行完后对结果集进行assertNotNull判断来实现 ASK 查询；

  • 路径查询：SPARQL 可以通过符号来替换冗余变量，笔者重写为Jena支持的格式来实现路径查询；比如s—p{m}-o，笔者就改写为完全展开；

重写的完整流程：

  1. 确定查询中where子句的内容；根据where来获取到 BGP（ basic graph pattern）
  2. 遍历 BGP set并进行重写；
  3. 确定当前 BGP 是否包含路径查询；否则 4，是则根据路径查询类型重写为等价语义无路径查询语句；重复当前步骤；
  4. 将重写后的 BGP 放入 rewritten collection；
  5. 从 rewritten collection获取无路径查询语句的where子句；

The Gremlin Graph Traversal Machine and Language

背景

论文给出了Gremlin的数学描述，并详细介绍了它的自动性和函数性；Gremlin主要由Graph、Traversal和Traverser组成；Graph就是点边图、Traversal就是一组指令，Traverser则是根据Traversal的指令在 Graph上游走；

介绍

The Graph

A traverser包含了以上六元组：

基于图查询系统的图计算引擎
背景

笔者涉及的是一个分布式图系统，他主要是通过对一些底层索引的数据结构设计，将图计算和图查询结合起来，同时也涉及到一些分布式场景下数据更新的实现，这块目前对我来说没什么参考价值，主要是看前半部分；

主要工作

图查询系统的哈希索引：

在笔者涉及的Wukong系统中，图上的边会转换成键值对进行存储，key：源顶点类型、边类型、边方向、值的地址/大小；value：邻居顶点（数组形式）；比如上图中的源顶点 1，边类型 2、边方向in就能找到它的领居 8、9；哈希索引在单点查询的效率是极高的；

图计算系统的稀疏矩阵：

主要有行压缩稀疏矩阵（compressed sparse row，CSR）表示出边的信息，列压缩稀疏矩阵（compressed sparse column，CSC）表示入边的信息；顶点索引（vertex index）记录了每个顶点在边数组中的起始位置，并且顶点编号与顶点索引数组的序号保持一致；如上图的顶点 0 没有邻居，顶点 4 对应邻居1,3；很明显这种数据结构在图计算时是非常高效，因为可以以极快的速度找到它对应的邻居；

笔者针对图计算的索引优化技术实现：

很巧妙地将前面提到的两种数据结构结合在起来；在新的存储模式下，为了便于索引的访问，值需要按顶点ID有序并且连续存储。但这样的限制不会对原先的图查询系统产生性能影响；具有很好的空间局部性；

收获
  • 笔者用到的数据集
  • 数据结构的设计，笔者没有谈及图更新操作；如果仅仅是图查询和图计算，这个数据结构设计得很漂亮，可以思考在自己课题设计底层存储时能否实现，不要直接使用关系型数据库来设计，看能否达到优化的效果；
Synergistic Graph and SQL Analytics Inside IBM Db2
背景

IBM Db2数据库，同时支持 SQL 和Graph query；因为历史原因，主要还是关系型存储；像sqlg这些，虽然是建立在关系型数据库之上，但是对格式有特殊要求；还有一些是将图从关系型数据库抽取出来放到内存中进行操作，显然这些需要维护数据的一致性；而本质上IBM Db2还是关系型存储，然后上层设计了Db2 Graph来支持图查询；即 SQL 和Graph操作的是同一份数据；

实现

图是属性图；Db2 Graph employs agraph overlay approach to expose agraph view of the relational data.很抽象的一句话；它是通过overlay configuration files，就是配置文件的方式，建立起关系型数据和图数据之间的映射关系；比如：哪个table 存边、那一列对应id；那一列对应 label等等；

系统架构：

Db2 Graph是介入Gremlin图查询和Db2查询引擎的层；top是tinkerpop，负责解析Gremlin生成tinkerpop的api调用；然后下面 4 层，负责实现tinkerpop core api和一些优化遍历策略；

  • topology：属性图和关系表的映射；
  • Graph Structure：实现tinkerpop graph structure API；
  • sql dialect（方言）：处理Db2特定的实现细节；
  • Traversal Strategy ：优化策略

两个重要优化：

  • Optimization for fixed labels：通过固定label来缩小查询范围；
  • Optimized traversal strategies：这个也是我要做的工作；可参考，但是感觉大家实现的都大差不差；都是通过合并一些单步等操作来提前过滤数据等；
收获
  • 图数据库：Neo4j、JanusGraph、Sqlg；
  • 可参考优化策略的实现；
