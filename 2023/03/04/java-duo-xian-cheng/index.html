Java线程

首先，明确一点，JVM是不具备创建线程的能力，是由操作系统来创建线程。
我们来看Thread类的源码，源码开头有一大段话：

从这段话可以提炼出以下几点：

  • JVM允许一个应用有多个线程同时运行；
  • 每个线程都用一个优先级，高优先级的线程会比低优先级的线程优先执行（即更有可能被操作系统的CPU调度执行）；新线程与创建它的线程的优先级一致；
  • 线程分为守护线程和非守护线程；当且仅当创建线程的线程是守护线程时，被创建的线程是守护线程；
  • 当JVM启动时，通常有一个非守护线程，该线程调用某个指定类名为Main()的方法；
  • 线程终止条件：（1）调用Runtime.getRuntime().exit(status)方法，安全管理器允许退出（2）所有非守护线程死亡：从run()方法返回或者抛出超出run()方法的异常，即run()方法不处理该异常或者说是在执行run()方法时出现了异常；

线程状态

Java层面，线程一共有6种状态；Thread类中有一个枚举类State进行了详细说明

public enum State {
        /**
         * Thread state for a thread which has not yet started.
         */
        NEW, // 线程还未开始；

        /**
         * Thread state for a runnable thread.  A thread in the runnable
         * state is executing in the Java virtual machine but it may
         * be waiting for other resources from the operating system
         * such as processor.
         */
        RUNNABLE, // 线程可以被JVM执行，但是在等待操作系统资源，如处理器资源；

        /**
         * Thread state for a thread blocked waiting for a monitor lock.
         * A thread in the blocked state is waiting for a monitor lock
         * to enter a synchronized block/method or
         * reenter a synchronized block/method after calling
         * {@link Object#wait() Object.wait}.
         */
        BLOCKED, // 等待monitor lock，即调用monitorenter导致的阻塞

        /**
         * Thread state for a waiting thread.
         * A thread is in the waiting state due to calling one of the
         * following methods:
         * <ul>
         *   <li>{@link Object#wait() Object.wait} with no timeout</li>
         *   <li>{@link #join() Thread.join} with no timeout</li>
         *   <li>{@link LockSupport#park() LockSupport.park}</li>
         * </ul>
         *
         * <p>A thread in the waiting state is waiting for another thread to
         * perform a particular action.
         *
         * For example, a thread that has called <tt>Object.wait()</tt>
         * on an object is waiting for another thread to call
         * <tt>Object.notify()</tt> or <tt>Object.notifyAll()</tt> on
         * that object. A thread that has called <tt>Thread.join()</tt>
         * is waiting for a specified thread to terminate.
         */
        WAITING, // 调用Object.wait()、Thread.join()、LockSupport.park()

        /**
         * Thread state for a waiting thread with a specified waiting time.
         * A thread is in the timed waiting state due to calling one of
         * the following methods with a specified positive waiting time:
         * <ul>
         *   <li>{@link #sleep Thread.sleep}</li>
         *   <li>{@link Object#wait(long) Object.wait} with timeout</li>
         *   <li>{@link #join(long) Thread.join} with timeout</li>
         *   <li>{@link LockSupport#parkNanos LockSupport.parkNanos}</li>
         *   <li>{@link LockSupport#parkUntil LockSupport.parkUntil}</li>
         * </ul>
         */
        TIMED_WAITING,

        /**
         * Thread state for a terminated thread.
         * The thread has completed execution.
         */
        TERMINATED;
    }

    线程方法

    Thread.yield()方法

public static native void yield(); // 静态方法


调用yield()方法表示当前线程愿意主动放弃处理器的使用权，但是调度器可以选择忽略。所以这个方法有点类似Thread.sleep()方法；但是它没有规定时间以及是否一定会释放CPU；它仅仅是表示可以把CPU的使用权让给优先级更高的线程，但是自己也会重新加入的CPU使用权的争取（可以理解为runnable状态）

Thread.sleep()方法

public static native void sleep(long millis) throws InterruptedException;  // 静态方法，会抛出中断异常

Thread.sleep()方法比较常见，就是让出CPU休眠固定时间，再重新争取CPU的使用权；需要注意的是，它会抛出中断异常；也就是说，线程在休眠期间是可以检测到中断的，即是其他线程调用了thread.interrupt()方法（thread表示休眠的线程）；sleep()方法会清除中断状态再抛出异常；所以我们通常会调用sleep()方法时使用try-catch，在catch里面选择（1）重新设置中断状态亦或是继续向上抛出异常。（2）别睡了，起来干活，这里就是根据具体场景来处理了。

thread.join()方法

public final synchronized void join(long millis)
throws InterruptedException {
    long base = System.currentTimeMillis();
    long now = 0;

    if (millis < 0) {
        throw new IllegalArgumentException("timeout value is negative");
    }

    if (millis == 0) {
        while (isAlive()) {
            wait(0);
        }
    } else {
        while (isAlive()) {
            long delay = millis - now;
            if (delay <= 0) {
                break;
            }
            wait(delay);
            now = System.currentTimeMillis() - base;
        }
    }
}

我们通常是用无参数的join()方法；如果是带时间参数，那就是当前线程调用后等待该时间，然后重新争取CPU；如果不带参数：thread.join()；就是等thread线程执行完之后，当前线程再继续执行。

public final void join() throws InterruptedException {
    join(0); // 0表示无限等待
}

这个方法也会抛出中断异常；是因为调用了wait()方法，如

while (isAlive()) {
    wait(0); // object.wait()
}

可以看出，join()方法join()方法带synchronized关键字；本质上就是先获得thread对象锁，判断线程是否存活isAlive()，还存活我就wait()；这里的wait()就是obeject.wait()方法，；这里面可能有一点点绕，我们用一个例子来看

public static void main(String[] args) throws InterruptedException {
    Thread thread = new Thread(() -> {
        for (int i = 0; i < 1000; i++) {
            System.out.println(i);
        }
    });
    thread.start();
    thread.join(); // main线程会等待thread线程执行完再继续执行
    System.out.println("Main");
}

通过这样来看的话，join()方法实际上是

public static void main(String[] args) throws InterruptedException {
    Thread thread = new Thread(() -> {
        for (int i = 0; i < 1000; i++) {
            System.out.println(i);
        }
    });
    thread.start();
    synchronized (thread) { // 线程对象作为锁对象
        while (thread.isAlive()) { // 判断存活
            thread.wait(0); // 存活则等待锁
        }
    }
    System.out.println("Main");
}

那么问题来了，thread.wait()之后，是怎么唤醒的呢。

注解里面说到，线程在执行完之后会调用notifyAll()方法。但是这个notifyAll()在Java层面是看不到的，别人的解释说是在JVM层面实现了。上面的注解也强调了其他场景不要用thread实例作为锁对象。

中断方法

线程中断方法一共有3个：

thread.interrupt()方法
public void interrupt() {
    if (this != Thread.currentThread())
        checkAccess(); // 判断是否具有修改当前线程的权限，无则抛异常，不是本文讨论的重点

    synchronized (blockerLock) {
        Interruptible b = blocker;
        if (b != null) {
            interrupt0();           // Just to set the interrupt flag
            b.interrupt(this);
            return;
        }
    }
    interrupt0();
}


注解里面说到，当被中断的线程阻塞在object.wait()，thread.join()，Thread.sleep()方法，就是说，线程在等待锁、在等待别的线程执行完、在休眠时被中断，就会清除中断标记（interrupt方法会把线程的中断标记设为true），抛出中断异常。也很绕，我们还是来看代码：

Thread.sleep()场景

public static void main(String[] args) {
    Thread thread = new Thread(() -> {
        try {
            Thread.sleep(10000);
        } catch (InterruptedException e) {
            System.out.println("InterruptedException"); // 休眠被中断，抛出中断异常
        }
    });
    thread.start();
    thread.interrupt();
}

object.wait()场景
public static void main(String[] args) {
    Object object = new Object();
    Thread t1 = new Thread(() -> {
        synchronized (object) {
            System.out.println("t1");
            try {
                object.wait();
            } catch (InterruptedException e) { // wait()被中断，抛出中断异常
                System.out.println("InterruptedException");
            }
        }
    });
    Thread t2 = new Thread(() -> {
        synchronized (object) {
            while (true) {
            }
        }
    });
    // 这里要注意t1不一定比t2先执行
    t1.start(); 
    // 这里可以休眠一会儿，保证t1先于t2执行
    t2.start();
    t1.interrupt();
}

thread.join()场景
public static void main(String[] args) {
    Thread t1 = new Thread(() -> {
        Thread t2 = new Thread(() -> {
            while (true) {
            }
        });
        t2.start();
        try {
            t2.join();
        } catch (InterruptedException e) {
            System.out.println("InterruptedException");
        }
        System.out.println("t1");
    });
    t1.start();
    t1.interrupt();
}

通过上面几种场景，可以看出Java线程之间不是一种强制式，而是一种协调式，就是说，当线程A想要中断线程B的执行，会被线程B打上中断标记，至于线程B要不要中断，就看你具体的实现了。像以上三种场景就是抛出中断异常，你可以通过捕获异常来进行相应的处理，比如说我不想中断，那我就可以继续重新调用相应的方法（通常不会这么干，因为代码都是自己写的，想要中断你肯定是有原因的）；那么除了以上的场景，其他场景就需要线程主动去检查是否有中断标记，如果线程不主动检查中断标记，你把它标烂了，它都不会鸟你。

Thread.interrupted()方法
public static boolean interrupted() {
    return currentThread().isInterrupted(true); // 清除中断标记
}

检测是不是有别的线程想要来中断我；直接看isInterrupted()方法就行

thread.isInterrupted()方法
public boolean isInterrupted() {
    return isInterrupted(false);
}
private native boolean isInterrupted(boolean ClearInterrupted);


看注解很容易理解，两个方法都是用来检测中断，默认的isInterrupted()方法不会清除中断标记，而interrupted()会清除中断标记。

public static void main(String[] args) throws InterruptedException {
    Thread t1 = new Thread(() -> {
        while (!Thread.interrupted()) { // 检测中断，并清除
            // do something
        }
        System.out.println(Thread.currentThread().isInterrupted()); // false
    });
    t1.start();
    Thread.sleep(3000);
    t1.interrupt();
}

所以，Java的中断线程并不是让一个线程停止运行，只是将线程的中断标志设为true, 或者在某些特定情况下抛出一个InterruptedException，并不会直接将一个线程停掉，也就是说，无论是设置中断标志还是抛出InterruptedException，它们都是给当前线程的建议，当前线程可以选择采纳或者不采纳，它们并不会影响当前线程的执行。

线程池执行步骤

ThreadPoolExecutor

顶级接口Executor，Executor就只有一个execute()方法，用来执行被提交的runnable任务；它把任务提交和任务运行分离；说白了就是你把runnable对象丢进来就行，不用关注任务具体是怎么执行的。

public interface Executor {

    /**
     * Executes the given command at some time in the future.  The command
     * may execute in a new thread, in a pooled thread, or in the calling
     * thread, at the discretion of the {@code Executor} implementation.
     *
     * @param command the runnable task
     * @throws RejectedExecutionException if this task cannot be
     * accepted for execution
     * @throws NullPointerException if command is null

}

Runnable对象，注释里说到，当一个实现接口Runnable的对象被用来创建一个线程时，启动线程会导致该对象的run方法在该单独执行的线程中被调用。就是我们常说的thread.start()会执行runnable对象的run()方法

public interface Runnable {
    /**
     * When an object implementing interface <code>Runnable</code> is used
     * to create a thread, starting the thread causes the object's
     * <code>run</code> method to be called in that separately executing
     * thread.
     * <p>
     * The general contract of the method <code>run</code> is that it may

注释中提到了3个类：

execute()方法

这里关于ThreadPoolExecutor的核心参数就不展开了，直接看execute()方法

execute()方法主要是判断线程池状态是否满足去执行addWorker()方法，addWorker()就是去创建工作线程，core参数用于区分创建核心线程和非核心线程；所以execute()方法就是把runnable任务添加到工作线程

   public void execute(Runnable command) { // 新来的任务调用excute()方法
        // 健壮性判断
        if (command == null)
            throw new NullPointerException();
        /*
         * Proceed in 3 steps:
         *
         * 1. If fewer than corePoolSize threads are running, try to
         * start a new thread with the given command as its first
         * task.  The call to addWorker atomically checks runState and
         * workerCount, and so prevents false alarms that would add
         * threads when it shouldn't, by returning false.
         *
         * 2. If a task can be successfully queued, then we still need
         * to double-check whether we should have added a thread
         * (because existing ones died since last checking) or that
         * the pool shut down since entry into this method. So we
         * recheck state and if necessary roll back the enqueuing if
         * stopped, or start a new thread if there are none.
         *
         * 3. If we cannot queue task, then we try to add a new
         * thread.  If it fails, we know we are shut down or saturated
         * and so reject the task.
         */
     		// 获取ctl变量：线程数和线程池状态
        int c = ctl.get();
     		// 判断当前工作线程数是否小于核心线程数，true则尝试加入任务并执行
        if (workerCountOf(c) < corePoolSize) {
          	// addWorker第二个参数为true，表示用核心线程执行任务，addWorker()方法返回true表示成功加入任务并执行；execute()方法结束返回
            if (addWorker(command, true)) // core参数为true，即创建核心线程来执行任务，其中创建核心线程addWorker和workcount计数是分开的，下面会讲
                return;
// --------------------------------------------------------------------------------------------
// 上面的步骤是最简单直接的，添加成功，直接完事了
          	// 如果false，则再次获取ctl，因为可能在调用addWorker()时候线程池状态发生了变化：前面的workerCountOf(c) < corePoolSize和addWorker(command, true)不是原子性操作
            c = ctl.get();
        }
     		// &&连接，线程池处于running的话，尝试把任务加入工作队列
        if (isRunning(c) && workQueue.offer(command)) { // private final BlockingQueue<Runnable> workQueue;
          	// 任务加入工作队列成功，再次获取ctl变量
            int recheck = ctl.get();
            // 如果线程池处于非running，移除任务这里是因为前面的 &&连接不是原子性操作，可能在执行workQueue.offer(command)操作时，线程池的状态改变了
            if (! isRunning(recheck) && remove(command))
                // 拒绝任务，抛出异常：handler.rejectedExecution(command, this);
                reject(command); // 就是非运行状态，不允许添加任务
// --------------------------------------------------------------------------------------------
// 上面的步骤就是核心线程满了，尝试将任务加入缓存队列
          	// 线程池处于running，但是没有工作线程，创建一个非核心线程来处理缓存队列的任务，这里主要有两种情况（1）corePoolSize设置为了0；（2）allowsCoreThreadTimeOut设置为true
            else if (workerCountOf(recheck) == 0)
                addWorker(null, false);
        }
     		// 缓存队列满了，直接就创建一个非核心线程执行任务
        else if (!addWorker(command, false))
            // 失败就拒绝
            reject(command);
    }

可以看到，线程池在工作线程未达到corePoolSize的时候，不会复用线程，只要有任务来就会创建工作线程；当核心工作线程都满了之后，对于新来的任务，会先加入阻塞队列（此时核心工作线程执行完任务后会从阻塞队列取任务来执行）；当阻塞队列也满了之后（说明核心工作线程忙不过来了），会尝试创建非核心线程来执行任务，如果非核心线程创建失败（达到maximumPoolSize），则执行拒绝策略。

addWorker()方法

execute()方法的核心是addWorker()方法；addWorker()方法一进去就有两层循环，经过了这两层循环的判断，才能真正进入添加工作线程的流程：

  • 外层循环是判断线程池状态，能否添加工作线程，添加工作线程就是创建worker对象，加入到线程池的HashSet\ workers里面；线程池处于以下两种状态可以添加工作线程（1）running状态（2）shutdown状态，此时不能添加新任务，但是缓存队列BlockingQueue\ workQueue不为空的话，允许添加工作线程；
  • 内层循环是判断线程池内工作线程的情况，是否允许添加工作线程，允许添加的话，workerCount通过CAS操作加1，break两层循环，进入添加工作线程的流程；这里workerCount + 1和workers.add(worker)不是原子性操作，是分开的，+1比较轻量，如果到时候添加工作线程失败了再-1；
  • 总结，两层for循环成功走完，得出一个结论：当线程池可以添加工作线程，只干了一件事：workerCount + 1
  • 不得不说这两层for循环写得很艺术，因为它并不是真的需要循环去做什么事情，因为循环体可能顶多就走一两次，它更多的是通过循环体来感知线程池的状态和重试
private boolean addWorker(Runnable firstTask, boolean core) { // core表示是否用核心线程执行任务
        // 类似goto语句，用在循环体之前
  			retry:
        // 外层循环判断线程池状态
        for (;;) {
            int c = ctl.get();
            int rs = runStateOf(c);

            // Check if queue empty only if necessary.
          	// rs >= SHUTDOWN，running为-1，直接false跳过了
            // ! (rs == SHUTDOWN && firstTask == null && ! workQueue.isEmpty())
          	// 上面这句话比较绕，等价于 rs > SHUTDOWN || firstTask != null || workQueue.isEmpty()
            // 上面三个条件任意满足其中一个就是返回false跳过了，就是说在shutdown以上状态，你最多只能执行队列里面的任务，不能再提交任务（这里就是避免饥饿，就是你是shutdown状态，你没有提交任务，但是你的队列里面还有任务，那么得允许去创建线程）
            if (rs >= SHUTDOWN &&
                ! (rs == SHUTDOWN &&
                   firstTask == null &&
                   ! workQueue.isEmpty()))
                return false;
            // 内层循环判断工作线程数
            for (;;) {
                int wc = workerCountOf(c);
                // CAPACITY是最大上限
                if (wc >= CAPACITY ||
                    wc >= (core ? corePoolSize : maximumPoolSize))
                    return false;
              	// CAS操作工作线程+1，这里只是计数加1，计算+1成功了直接跳出retry紧跟的循环体，跳至下面真正创建线程的代码段
                if (compareAndIncrementWorkerCount(c))
                    break retry;
                // 如果计数加1没有成功，重新判断线程池状态，基本上就相当于继续执行+1操作
                c = ctl.get();  // Re-read ctl
                if (runStateOf(c) != rs)
                    continue retry; // 线程池非running状态，直接重新执行retry紧跟的循环体
                // else CAS failed due to workerCount change; retry inner loop
            }
        }
				// 真正创建工作线程的过程
        boolean workerStarted = false;
        boolean workerAdded = false;
        Worker w = null; // worker对象有final Thread thread成员变量，用来真正执行task
        try {
            w = new Worker(firstTask); // 这里会初始化thread成员变量，getThreadFactory().newThread(this)；this就是新建的worker，所以worker的thread执行的就是worker的run（）方法，而worker的run()调用了ThreadPoolExecutor的runWorker(this)，this就是新建的worker；既然是创建工作线程，肯定是要new一个worker对象，然后把任务丢进去，这个丢进去的任务很显然就是worker要直接处理的任务，如果firstTask为空，则是去缓存队列里面取任务，后面的runworker()方法会详细讲；注意这里firstTask可能是null
            final Thread t = w.thread;
            if (t != null) { // 因为线程是由ThreadFactory创建，存在自定义ThreadFactory导致线程为null
                // 加锁，操作workers
                final ReentrantLock mainLock = this.mainLock;
                mainLock.lock();
                try {
                    // Recheck while holding lock.
                    // Back out on ThreadFactory failure or if
                    // shut down before lock acquired.
                    int rs = runStateOf(ctl.get());
                  
                    if (rs < SHUTDOWN ||
                        (rs == SHUTDOWN && firstTask == null)) { // （1）运行状态可以添加worker，（2）shutdown状态，添加worker处理缓存队列任务，不再接收新任务
                        if (t.isAlive()) // precheck that t is startable isAlive()方法Tests if this thread is alive. A thread is alive if it has been started and has not yet died.
                            throw new IllegalThreadStateException(); // 简单的异常处理，因为正常情况下线程肯定不是运行态
                        workers.add(w); // private final HashSet<Worker> workers = new HashSet<Worker>(); 加入缓存队列，mainLock就是为了hashset的并发，workers.add(w)方法仅在这里调用
                        int s = workers.size();
                        if (s > largestPoolSize) // private int largestPoolSize，int类型，默认初始化为0
                            largestPoolSize = s; // 仅仅是记录线程池最大线程数，没什么特别含义
                        workerAdded = true;
                    }
                } finally {
                    mainLock.unlock(); // largestPoolSize只在mainLock里面赋值，保证准确性
                }
                if (workerAdded) { // 添加成功，调用start()方法执行任务，即启动新线程
                    t.start(); // 执行run()方法，前面说到线程t传入了worker对象，所以执行的是worker对象的run()方法，worker对象的run()方法执行runWorker(this)
                    workerStarted = true;
                }
            }
        } finally { // 为什么需要workerAdded和workerStarted两个变量，t.start()会抛异常？
            if (! workerStarted)
                addWorkerFailed(w); // 添加线程失败，恢复计数，什么情况下会走到这一步？（1）t有问题，（2）线程池状态有问题
        }
        return workerStarted;
    }

我们忽略addWorker()方法前半部分的判断逻辑，关注创建worker的过程：

单纯看这部分逻辑是比较简单的，就是先加锁，因为涉及到HashSet\ workers；然后在里面再一次判断是否满足创建工作线程的条件，满足就创建并执行任务，线程的start()只能执行一次，真正对线程的复用是在run()方法里面实现的。

// addWorker添加线程失败执行addWorkerFailed方法，（1）从工作线程队列移除该worker；（2）workerCount - 1；（3）尝试关闭线程池
    private void addWorkerFailed(Worker w) { // 但是最大线程数没有被恢复
        final ReentrantLock mainLock = this.mainLock;
        mainLock.lock(); // 涉及workers，加锁操作
        try {
            if (w != null)
                workers.remove(w); // 从缓存队列移除添加失败的worker
            decrementWorkerCount(); // 恢复计数
            tryTerminate(); // 为什么就关闭线程池
        } finally {
            mainLock.unlock();
        }
    }

final void tryTerminate() {
        for (;;) {
            int c = ctl.get();
            if (isRunning(c) ||
                runStateAtLeast(c, TIDYING) ||
                (runStateOf(c) == SHUTDOWN && ! workQueue.isEmpty()))
                return;
            if (workerCountOf(c) != 0) { // Eligible to terminate
                interruptIdleWorkers(ONLY_ONE);
                return;
            }

            final ReentrantLock mainLock = this.mainLock;
            mainLock.lock();
            try {
                if (ctl.compareAndSet(c, ctlOf(TIDYING, 0))) {
                    try {
                        terminated();
                    } finally {
                        ctl.set(ctlOf(TERMINATED, 0));
                        termination.signalAll();
                    }
                    return;
                }
            } finally {
                mainLock.unlock();
            }
            // else retry on failed CAS
        }
    }
private final class Worker
    extends AbstractQueuedSynchronizer
    implements Runnable  // ThreadPoolExecutor内部类，实现runnable接口
{
    /** Thread this worker is running in.  Null if factory fails. */
    final Thread thread;
    /** Initial task to run.  Possibly null. */
    Runnable firstTask;
    /** Per-thread task counter */
    volatile long completedTasks;

    /**
     * Creates with given first task and thread from ThreadFactory.
     * @param firstTask the first task (null if none)
     */
    Worker(Runnable firstTask) {
        setState(-1); // inhibit interrupts until runWorker
        this.firstTask = firstTask;
        this.thread = getThreadFactory().newThread(this);
    }

    /** Delegates main run loop to outer runWorker  */
    public void run() {
        runWorker(this); // ThreadPoolExecutor的方法
    }
}

runWorker()方法

前面讲到addWorker()方法满足条件就会调用t.start()方法去执行任务。

    Worker(Runnable firstTask) {
        setState(-1); // inhibit interrupts until runWorker
        this.firstTask = firstTask;
        this.thread = getThreadFactory().newThread(this);
    }

public void run() {

    }

从上面可以看出，把worker对象丢进线程，worker实现了runnable接口，线程执行的就是worker的run()方法，也是线程池的runWorker(worker)方法。runWorker()方法的while循环条件不成立，线程就执行结束了。

final void runWorker(Worker w) {
        Thread wt = Thread.currentThread();
  			// 线程只有在第一次执行的时候可能带有firstTask，后面都是通过getTask()方法获取任务
        Runnable task = w.firstTask;
        w.firstTask = null;
        w.unlock(); // allow interrupts，// 因为worker对象初始化时setState(-1)，worker继承了AQS，unlock()实际上是调用trytryRelease()方法setState(0);这意味着只有调用了runWorker方法的线程才能被中断
        boolean completedAbruptly = true;
        try {
            while (task != null || (task = getTask()) != null) {
                w.lock(); // 这里加锁了，意味着如果任务在运行，shutdown()方法是没法中断的，因为shutdown()方法会先tryLock()
                // If pool is stopping, ensure thread is interrupted;
                // if not, ensure thread is not interrupted.  This
                // requires a recheck in second case to deal with
                // shutdownNow race while clearing interrupt
                if ((runStateAtLeast(ctl.get(), STOP) ||
                     (Thread.interrupted() &&
                      runStateAtLeast(ctl.get(), STOP))) &&
                    !wt.isInterrupted())
                    //第一种情况： (runStateAtLeast(ctl.get(), STOP) && !wt.isInterrupted()
                    // 如果是stop以上状态
                    // 第二种情况：如果已经有中断标记（清除），再次判断是否为stop以上状态
                    // 说实话，没太看明白，可能是指令重排导致先打中断再设置线程池状态为stop？
                    wt.interrupt(); // 这里的中断并不是一定能实现，主要看我们的task.run()任务里面有没有可以被中断的条件，通常情况下我们的实现run方法体都会允许中断
                try {
                   // 空方法，类似代理模式，允许用户自行扩展，面向切面编程
                    beforeExecute(wt, task);
                    Throwable thrown = null;
                    try {
                        task.run();
                    } catch (RuntimeException x) {
                        thrown = x; throw x;
                    } catch (Error x) {
                        thrown = x; throw x;
                    } catch (Throwable x) {
                        thrown = x; throw new Error(x);
                    } finally {
                        afterExecute(task, thrown);
                    }
                } finally {
                    task = null;
                    w.completedTasks++;
                    w.unlock();
                }
            }
            completedAbruptly = false;
        } finally {
            processWorkerExit(w, completedAbruptly); // 资源清理，从HashSet移除worker，JVM得以回收；
        }
    }

getTask()方法

getTask()方法跟前面那些方法很类似，都是先判断状态，再获取任务

private Runnable getTask() {
        boolean timedOut = false; // Did the last poll() time out?

        for (;;) {
            int c = ctl.get();
            int rs = runStateOf(c);

            // Check if queue empty only if necessary.
            // （1）线程池为shutdown以上状态（2）shutdown状态但是队列为空，以上两种情况可以直接销毁线程了
          	// 这里如果是stop状态，直接返回null了，就是不处理了；如果是shutdown的话，有任务就处理，没任务也直接返回了，不会走到下面的阻塞
            if (rs >= SHUTDOWN && (rs >= STOP || workQueue.isEmpty())) {
                // do {} while (! compareAndDecrementWorkerCount(ctl.get()));
                decrementWorkerCount(); // workerCount-1，销毁当前worker
                return null;
            }

            int wc = workerCountOf(c);

            // Are workers subject to culling?
            // 线程池内部本质上是没有区分核心和非核心线程，通过timed判断该worker是否可以销毁
            boolean timed = allowCoreThreadTimeOut || wc > corePoolSize;
          
						// (wc > maximumPoolSize || (timed && timedOut))销毁当前worker的两个前提条件（1）超过最大线程数，（2）超时；在满足这两个前提条件下，如果wc > 1直接销毁当前worker，因为还有其他worker还可以处理任务，或者是缓存队列为空，也可销毁 
            if ((wc > maximumPoolSize || (timed && timedOut))
                && (wc > 1 || workQueue.isEmpty())) {
                if (compareAndDecrementWorkerCount(c))
                    return null;
                continue; // 这里的continue感觉就是重试的意思
            }

            try {
                Runnable r = timed ?
                    workQueue.poll(keepAliveTime, TimeUnit.NANOSECONDS) :
                    workQueue.take(); // 这个线程不可销毁，尬等着，就摁等到任务或者中断发生
                if (r != null)
                    return r;
                timedOut = true; // 超时了
            } catch (InterruptedException retry) {
                timedOut = false;
            }
        }
    }

shutdown()方法

public void shutdown() {
    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock();
    try {
        checkShutdownAccess();
        advanceRunState(SHUTDOWN);
        interruptIdleWorkers();
        onShutdown(); // hook for ScheduledThreadPoolExecutor
    } finally {
        mainLock.unlock();
    }
    tryTerminate();
}

关键在interruptIdleWorkers()方法

private void interruptIdleWorkers() {
    interruptIdleWorkers(false);
}

private void interruptIdleWorkers(boolean onlyOne) {
    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock(); // 锁重入
    try {
        for (Worker w : workers) { // 遍历workers
            Thread t = w.thread; // 获取worker中的thread
            // 这里是tryLock()，也是在运行中的线程不会被打上中断标记，这些线程将会跑完线程池里面的任务，那么这些线程会在什么时候结束呢？getTask()方法中，如果线程状态不在是运行态，阻塞队列也没任务，直接返回null，那么runWorker()方法的while循环就结束了，线程也就结束了。
            if (!t.isInterrupted() && w.tryLock()) { // 如果t没有被中断并且w可以获取锁，可以获取锁说明线程不是处于运行态
                try {
                    t.interrupt(); // 中断线程
                } catch (SecurityException ignore) {
                } finally {
                    w.unlock(); // 释放锁
                }
            }
            if (onlyOne)
                break;
        }
    } finally {
        mainLock.unlock();
    }
}

这样打上中断标记后，在等待任务的线程就会被中断了；因为空闲线程会在调用workQueue.take()方法阻塞，该方法会抛出中断异常，捕获该异常就可以停止空闲线程了。

shutdownNow()方法

shutdownNow()方法更加粗暴，它不会去获取锁，直接就中断。

public List<Runnable> shutdownNow() {
    List<Runnable> tasks;
    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock();
    try {
        checkShutdownAccess();
        advanceRunState(STOP);
        interruptWorkers();
        tasks = drainQueue(); // shutdownNow()方法还会返回未执行的任务
    } finally {
        mainLock.unlock();
    }
    tryTerminate();
    return tasks;
}
private void interruptWorkers() {
    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock();
    try {
        for (Worker w : workers) // 没有获取锁的操作
            w.interruptIfStarted();
    } finally {
        mainLock.unlock();
    }
}

private List<Runnable> drainQueue() {
    BlockingQueue<Runnable> q = workQueue;
    ArrayList<Runnable> taskList = new ArrayList<Runnable>();
    q.drainTo(taskList);  // Removes all available elements from this queue and adds them to the given collection.
    if (!q.isEmpty()) {
        for (Runnable r : q.toArray(new Runnable[0])) { // 对特殊队列进行处理：有一些特殊的队列例如DelayQueue是无法立即通过drainTo方法将任务进行转移的，因此可能需要通过remove方法尝试将这些任务强制删除然后转移到新队列中。
            if (q.remove(r))
                taskList.add(r);
        }
    }
    return taskList;
}

真正打上中断的是在interruptIfStarted()方法

void interruptIfStarted() {
    Thread t;
    if (getState() >= 0 && (t = thread) != null && !t.isInterrupted()) { // getState() >= 0这里也对应着，只有进入了runWorker方法才允许中断
        try {
            t.interrupt();
        } catch (SecurityException ignore) {
        }
    }
}

区别

  1. shutdown()方法只能给在等待任务的线程打上中断标记（因为它有尝试获取锁的操作，在执行任务的线程持有锁）；shutdownNow()方法不需要获取锁，会给所有线程打上中断标记；
  2. shutdown()方法还可以处理阻塞队列里面的任务；而shutdownNow()不会处理阻塞队列里面的任务，而是通过一个List\列表将任务返回。（在getTask()方法中对线程池状态进行判断，如果是stop状态，会直接返回了，如果是shutdown状态，则工作线程可以继续取任务执行，而这些工作线程就是之前shutdown方法没有打上中断标记的线程）；
  3. shutdown()方法因为无法获取锁，所以不会中断正在执行的任务；而shutdownNow()方法会尝试中断正在执行的任务，这个尝试的含义就是说给线程打上了中断标记，但是不一定可以中断，主要看task.run()方法体，也就是我们提交的任务是否可以被中断（可以回去看上文提到过了中断场景）。

使用完线程池要调用shutdown()或shutdownNow()方法

我们通过前面源码分析可以知道；线程池是通过管理workers集合，然后真正执行任务的是worker的成员属性thread，thread执行runWorker()方法把worker作为参数传入，然后执行worker中的task，firstTask执行完之后会一直在getTask()，即使不再用任务传入，线程也会一直在阻塞。thread.start()会新起虚拟机栈，作为GC Root，如果不关闭线程池，资源会一直无法被回收，worker也无法被回收，worker对象作为线程池的内部类，所以线程池对象也无法被回收。
runWorker()方法执行完会调用processWorkerExit()方法，将worker从线程池的集合中移除。

private void processWorkerExit(Worker w, boolean completedAbruptly) {
    if (completedAbruptly) // If abrupt, then workerCount wasn't adjusted
        decrementWorkerCount();

    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock();
    try {
        completedTaskCount += w.completedTasks;
        workers.remove(w);
    } finally {
        mainLock.unlock();
    }

    tryTerminate();

    int c = ctl.get();
    if (runStateLessThan(c, STOP)) {
        if (!completedAbruptly) {
            int min = allowCoreThreadTimeOut ? 0 : corePoolSize;
            if (min == 0 && ! workQueue.isEmpty())
                min = 1;
            if (workerCountOf(c) >= min)
                return; // replacement not needed
        }
        addWorker(null, false);
    }
}

SpringBoot线程池

SpringBoot线程池的通常用法是先定义一个config类，在config类里面创建一个ThreadPoolTaskExecutor的Bean对象；而ThreadPoolTaskExecutor底层的实现其实是ThreadPoolExecutor：

@Configuration
@EnableAsync
public class TaskExecutorConfig {
	private static final String THREAD_NAME_PREFIX = "Task-";
    private int corePoolSize;
    private int maxPoolSize;
    private int queueCapacity;
    private int keepAliveSeconds;
    private final AtomicInteger threadId = new AtomicInteger(1);

    @Bean(name = "taskExecutor")
    @Primary
    public ThreadPoolTaskExecutor taskExecutor() {
        ThreadPoolTaskExecutor executor = new TaskExecutor(); // TaskExecutor类继承ThreadPoolTaskExecutor自定义了一些操作
        executor.setMaxPoolSize(maxPoolSize);
        executor.setCorePoolSize(corePoolSize);
        executor.setQueueCapacity(queueCapacity);
        executor.setKeepAliveSeconds(keepAliveSeconds);
        executor.setThreadFactory(new CustomizableThreadFactory() {
            @Nonnull
            @Override
            public Thread newThread(@Nonnull Runnable runnable) {
                String threadName = THREAD_NAME_PREFIX + threadId.getAndIncrement();
                return new Thread(getThreadGroup(), runnable, threadName);
            }
        });
        executor.setRejectedExecutionHandler(new AsyncRejectedExecutionHandler());
        executor.initialize();
        return executor;
    }

对ThreadPoolTaskExecutor 设置了相关的参数之后，调用executor.initialize()初始化线程池：

protected ExecutorService initializeExecutor(ThreadFactory threadFactory, RejectedExecutionHandler rejectedExecutionHandler) {
    BlockingQueue<Runnable> queue = this.createQueue(this.queueCapacity);
    ThreadPoolExecutor executor;
    if (this.taskDecorator != null) {
        executor = new ThreadPoolExecutor(this.corePoolSize, this.maxPoolSize, (long)this.keepAliveSeconds, TimeUnit.SECONDS, queue, threadFactory, rejectedExecutionHandler) {
            public void execute(Runnable command) {
                Runnable decorated = ThreadPoolTaskExecutor.this.taskDecorator.decorate(command);
                if (decorated != command) {
                    ThreadPoolTaskExecutor.this.decoratedTaskMap.put(decorated, command);
                }

                super.execute(decorated);
            }
        };
    } else {
    	// 底层实现是ThreadPoolExecutor
        executor = new ThreadPoolExecutor(this.corePoolSize, this.maxPoolSize, (long)this.keepAliveSeconds, TimeUnit.SECONDS, queue, threadFactory, rejectedExecutionHandler);
    }

    if (this.allowCoreThreadTimeOut) {
        executor.allowCoreThreadTimeOut(true);
    }

    this.threadPoolExecutor = executor;
    return executor;
}

可以看到，initialize()方法的实现initializeExecutor()方法底层是ThreadPoolExecutor。
