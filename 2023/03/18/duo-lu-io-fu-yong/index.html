多路IO复用

Java IO读写原理，我觉得这篇文章讲得很透彻了，按需搬运。

服务端和客户端通信过程：通常是服务服务端监听某个端口，有客户端请求到达，两者通过socket进行数据交互；

// 单线程同步阻塞IO
public void server1() {
    try {
        ServerSocket serverSocket = new ServerSocket(8080);
        while (true) {
            Socket socket = serverSocket.accept(); // 阻塞在这里，等待客户端连接
            handleSocket(socket);
        }
    } catch (IOException e) {
        e.printStackTrace();
    }
}

以上的实现方法，每次只能处理一个客户端请求，只有客户端请求完成了才继续调用accept()方法获取下一个请求；效率很低

// 多线程同步阻塞IO,线程多了就炸了
public void server2() {
    try {
        ServerSocket serverSocket = new ServerSocket(8080);
        while (true) {
            Socket socket = serverSocket.accept(); // 阻塞在这里，等待客户端连接
            new Thread(() -> handleSocket(socket)).start();
        }
    } catch (IOException e) {
        e.printStackTrace();
    }
}

以上的实现方法是，每个客户端socket都新建一个线程来处理，效率提高了，但是要一直创建线程和销毁线程；而且并发高了会导致资源耗尽；

// 线程池同步阻塞IO
public void server3() {
    ExecutorService executorService = Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors() * 2);
    try {
        ServerSocket serverSocket = new ServerSocket(8080);
        while (true) {
            Socket socket = serverSocket.accept(); // 阻塞在这里，等待客户端连接
            executorService.execute(() -> handleSocket(socket));
        }
    } catch (IOException e) {
        e.printStackTrace();
    }
}

以上的实现方法是使用线程池管理线程。

private void handleSocket(Socket socket) {
    InputStream inputStream;
    try {
        inputStream = socket.getInputStream();

        byte[] buffer = new byte[1024];
        int len;
        // 读取客户端数据
        while ((len = inputStream.read(buffer)) > 0) {
            System.out.println(new String(buffer, 0, len));
        }
        // 向客户端写数据
        OutputStream outputStream = socket.getOutputStream();
        outputStream.write("bye".getBytes());
    } catch (IOException e) {
        throw new RuntimeException(e);
    }
}

上面的实现中，很关键的一个步骤就是，调用accept()方法之后，服务端就会一直阻塞直到有数据。

我们来理解一下阻塞跟异步

阻塞

阻塞IO，指的是需要内核IO操作彻底完成后，才返回到用户空间，执行用户的操作。阻塞指的是用户空间程序的执行状态，用户空间程序需等到IO操作彻底完成。传统的IO模型都是同步阻塞IO。在java中，默认创建的socket都是阻塞的。

非阻塞IO，指的是用户程序不需要等待内核IO操作完成后，内核立即返回给用户一个状态值，用户空间无需等到内核的IO操作彻底完成，可以立即返回用户空间，执行用户的操作，处于非阻塞的状态。

所以，阻塞就是在内核和用户空间数据读写的时候，用户程序需要等待完成；

异步

同步IO是指用户空间线程是主动发起IO请求的一方，内核空间是被动接受方。

异步IO是指内核kernel是主动发起IO请求的一方，用户线程是被动接受方。

很抽象，不理解，先不管（因为好像现在异步IO不好用，主要还是用同步非阻塞IO）

所以，下面我们不再关注同步和异步，主要关注阻塞和非阻塞

IO多路复用

Java NIO浅析

IO多路复用是使用一个进程监视多个文件描述符，一旦某个描述符就绪（一般是内核缓冲区可读/可写），内核kernel能够通知程序进行相应的IO系统调用。

我们来看看Java层面的IO多路复用：

主要用三大组件：selector、channel、buffer；

之前我们在读写数据的时候，都是基于stream，在NIO中，是基于channel和buffer，buffer就是缓冲区，用户存放数据，channel是双向通道（不区分读写input/output）用于数据流通；buffer中的数据通过channel到达目标。

因为这个不太熟练，所以整个代码例子加深一下印象：

public static void main(String[] args) throws IOException {
    RandomAccessFile f = new RandomAccessFile("test.txt", "rw");
    FileChannel channel = f.getChannel();
    ByteBuffer wb = ByteBuffer.wrap("output".getBytes());
    // 将wb中的内容写入文件
    channel.write(wb);
    
    System.out.println(channel.position());
    channel.position(0); // 重置位置再进行读取

    ByteBuffer rb = ByteBuffer.allocate(128);
    // 文件内容读入rb
    channel.read(rb);
    // 翻转
    rb.flip();
    System.out.println(new String(rb.array(), 0, rb.remaining()));
}

然后就是selector和channel了：将channel注册到selector，当被监听的channel上有具体的事件发生时，selector发送通知，用户程序实现对相应事件的处理。

public static void main(String[] args) throws IOException {
        // 打开服务器套接字通道
        ServerSocketChannel serverSocketChannel = ServerSocketChannel.open();
        // 打开Selector
        Selector selector = Selector.open();
        // 服务器配置为非阻塞
        serverSocketChannel.configureBlocking(false);
        // 进行服务的绑定
        serverSocketChannel.bind(new InetSocketAddress(8080));
        // 将serverSocketChannel注册到selector，监听accept事件，即服务端收到客户端的连接
        serverSocketChannel.register(selector, SelectionKey.OP_ACCEPT);

        while (!Thread.currentThread().isInterrupted()) {
            selector.select();
            Set<SelectionKey> keys = selector.selectedKeys();
            Iterator<SelectionKey> keyIterator = keys.iterator();
            while (keyIterator.hasNext()) {
                SelectionKey key = keyIterator.next();
                if (key.isAcceptable()) {
                    // 接收客户端请求，建立连接，并将socketChannel注册到selector，监听读就绪事件
                    SocketChannel socketChannel = serverSocketChannel.accept();
                    socketChannel.configureBlocking(false);
                    socketChannel.register(selector, SelectionKey.OP_READ);
                } else if (key.isReadable()) {
                    // 监听到客户端可读，读就绪事件
                    SocketChannel socketChannel = (SocketChannel) key.channel();
                    ByteBuffer byteBuffer = ByteBuffer.allocate(128);
                    // 通过channel把数据读入buffer
                    socketChannel.read(byteBuffer);
                    byteBuffer.flip();
                    System.out.println(new String(byteBuffer.array(), 0, byteBuffer.remaining()));
                } else if (key.isWritable()) {
                    // 监听到客户端可写，写就绪事件
                    SocketChannel channel = (SocketChannel) key.channel();
                    ByteBuffer byteBuffer = ByteBuffer.wrap("output".getBytes());
                    // 将buffer内容通过channel写到客户端
                    channel.write(byteBuffer);
                    channel.register(selector, SelectionKey.OP_READ);
                }
                // 事件处理完毕，可以丢弃
                keyIterator.remove();
            }
        }
    }

接着我们来看Linux层面的select()/poll()/epoll()方法

select()/poll()/epoll()都是Linux实现IO多路复用的机制。

select()
int select(int nfds, fd_set *readfds, fd_set *writefds, fd_set *exceptfds, struct timeval *timeout);

select 函数监视的文件描述符分3类，分别是writefds、readfds、和exceptfds。调用后select函数会阻塞，直到有描述副就绪（有数据 可读、可写、或者有except），或者超时（timeout指定等待时间，如果立即返回设为null即可），函数返回。当select函数返回后，可以 通过遍历fdset，来找到就绪的描述符。

select目前几乎在所有的平台上支持，其良好跨平台支持也是它的一个优点。select的一 个缺点在于单个进程能够监视的文件描述符的数量存在最大限制，在Linux上一般为1024，可以通过修改宏定义甚至重新编译内核的方式提升这一限制，但 是这样也会造成效率的降低。

poll()
int poll (struct pollfd *fds, unsigned int nfds, int timeout);

不同与select使用三个位图来表示三个fdset的方式，poll使用一个 pollfd的指针实现。

struct pollfd {
    int fd; /* file descriptor */
    short events; /* requested events to watch */
    short revents; /* returned events witnessed */
};

pollfd结构包含了要监视的event和发生的event，不再使用select“参数-值”传递的方式。同时，pollfd并没有最大数量限制（但是数量过大后性能也是会下降）。 和select函数一样，poll返回后，需要轮询pollfd来获取就绪的描述符。

select()和poll()方法的主要区别就是最大数量限制。

epoll()
int epoll_create(int size)；//创建一个epoll的句柄，size为监听fd的数量
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event)；
int epoll_wait(int epfd, struct epoll_event * events, int maxevents, int timeout);

epoll的工作原理可以概括为：

  1. 创建一个epoll对象，通过epoll_create函数进行创建。
  2. 向epoll对象中添加需要监听的文件描述符，通过epoll_ctl函数进行添加。
  3. 调用epoll_wait函数等待事件的发生，当事件发生时，epoll_wait会返回一个包含事件信息的结构体数组。
  4. 遍历事件信息结构体数组，根据events字段判断发生的事件类型，执行相应的操作。
struct epoll_event {
  __uint32_t events;  /* Epoll events */
  epoll_data_t data;  /* User data variable */
};

epoll通过红黑树和链表两种数据结构来实现高效的I/O事件监听和事件发生的通知。

在epoll内部，它使用红黑树来管理需要监听的文件描述符，这个红黑树中的每个节点代表一个文件描述符，节点的key是文件描述符的值，节点的value是一个内部结构体，存储了这个文件描述符的状态和其他信息。这样可以使得epoll的事件注册和删除等操作的时间复杂度为O(log n)。

另外，epoll还使用一个双向链表来存储所有需要返回给用户的事件，这个链表称为就绪列表。当epoll监听到文件描述符上有事件发生时，会将事件对应的内部结构体添加到就绪列表中，然后通知用户进程有事件发生。在用户进程调用epoll_wait函数时，epoll会将就绪列表中的事件返回给用户进程，这样可以避免遍历所有文件描述符来查找有事件发生的描述符，从而提高epoll的效率。

综上所述，红黑树和链表是epoll实现高效的I/O事件监听和事件通知的关键数据结构。红黑树用于高效地管理需要监听的文件描述符，链表则用于存储就绪的事件，并且避免遍历所有文件描述符来查找有事件发生的描述符，从而提高epoll的效率。
