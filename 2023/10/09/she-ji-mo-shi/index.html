设计模式

无标签
发布日期:   2023-10-09
文章字数:   4.1k
阅读时长:   14 分

记住一句话：当你的代码里面出现冗余，那就是没有做好设计。

什么是设计模式

解决某一类问题的套路/方法；通常情况下我们直接去摁看设计模式的话，效果是很差的，我们更多地是要在具体的项目实践中，尤其是真正的项目，而不是练手的项目，你自己被真正的业务需求给恶心到了，你就会明白设计模式的重要性；而且设计模式也是从实践中提炼出来了。我用过的一些设计模式都是在实际开发中，发现把某些部分抽象一下、拆分一下，整个代码迭代扩展/测试起来变得方便很多，然后再去调研一下，自己用的某种设计模式的变体，或者说阉割版。

总结一句话就是，纸上得来终觉浅，绝知此事须躬行

UML

设计模式通常是表现类和类、类和接口之间的关系，所以我们要画图，画图就用到 UML，统一建模语言，关系主要包含：继承（三角形实线，子类指向父类）、实现（三角形虚线，实现类指向接口）、聚合（箭头被持有实线指向菱形持有）、关联（实现箭头，A 某种行为 B，如使用、创造、通知等行为）；除了以上4 个之外，还有泛化（就是继承）、组合等关系，但我看《图解设计模式》这本书，好像前 34种就够用了，这里就不展开了。

下面这张图包含了基本的关系，但是有些画的不太对，比如大雁和雁群的聚合关系。

我们常说的组合和聚合的区别，组合：不可独立存在，聚合：可以独立存在

掌握 UML 的好处就是，你看设计模式比较好懂，便于理解和记忆，毕竟图片比文字印象深刻多了。

除了 UML，我们还有一些时序图（系统各个模块/方法之间的调用顺序），状态图（比如写 Java 的同学会经常看到 Java 线程状态转换图），活动图（就是开始、结束、各种分支条件判断），泳道图（把不同的角色划分为不同的泳道，每个泳道有自己的职责，彼此之间可能存在交互）

设计模式

设计模式主要分为 3 大类：

  • 创建型模式：
  • 结构型模式
  • 行为型模式

迭代器模式

迭代器模式的作用就是不需要关注对象集合的实现方式，只要实现了迭代器接口，底层数据结构变化不影响迭代访问，如从[]数组变为ArrayList；迭代器遍历访问的逻辑不需要改动；

  • Aggreate接口包含iterator方法用来创建迭代器
  • Iterator接口包含遍历需要用到的hasNext和next方法
  • ConcreteAggreate就是具体的集合类，实现Aggreate接口中的iterator方法，如下
public Iterator iterator() {
    return new BookShelfIterator(this);
}
  • ConcreteIterator类构造方法接收ConcreteAggreate作为成员变量，实现hasNext和next方法遍历ConcreteAggreate，但是我看真正的迭代器实现，好像还有 remove 方法。而hasNext和next方法的实现依赖ConcreteAggreate，如hasNext的判断依赖Aggreate的size/length；next的获取依赖Aggreate的get(index)方法

以上就是迭代器模式的实现，如果ConcreteAggreate的数据结构发生了改变，我们需要做的就是去ConcreteAggreate的size/length和get(index)方法。这种实现的好处是：通过引入iterator，对于ConcreteAggreate的调用方，是不需要感知ConcreteAggreate的数据结构变更的，达到复用/最小修改的效果。

疑问：

  • 为什么要引入ConcreteIterator？ConcreteAggreate直接完成ConcreteIterator的工作不可行吗？好像这样的话，就需要用内部类来实现了。因为你肯定是要实现Iterator接口的，就看在哪里实现而已。
  • 为什么要引入Iterator？而不是直接使用ConcreteIterator？我的理解是（1）Aggreate和Iterator是顶级接口，不只是服务于ConcreteAggreate，而是为所有可遍历对象提供Iterator，所以引入Iterator是必要的；（2）依赖于抽象而不是具体，低耦合；

缺点：

  • 明显需要的资源更多了，简单的遍历多了好几个接口和类对象，多层调用效率肯定也有所下降
  • 不易理解，迭代器没有直接的for循环来的容易理解：next，hasNext概念

适配器模式

适配器模式主要作用是兼容，client调用 target，target 想复用已有类的能力，那么就可以在中间加一个适配器，target 调用adpater，adpater再去调用已有类的能力；兼容，复用有利于快速定位问题，因为问题大概率不会出现在已有的类里面。

适配器老是记不住，现在记住了，核心的一点是，client 直接使用的是适配器。

适配器模式有两种：

  • 类适配器（继承）
  • 对象适配器（委托）
类适配器

它们老是喜欢拿电压来举例，我感觉也不是很好理解；适配说白了就是一个中间转换器，client 想调用 A， 但我现在有的是 B，那就中间加个适配器（实现 A，调用 B）；即适配 B 类

继承的好处是，他不仅可以提高 client 想要的 A 能力，同时也具备了原本的 B 能力

另外就是当 A 是一个接口的时候，我们可以用继承来干。

对象适配器

如果 A 是一个抽象类的话，我们就不能使用类适配器了，因为 Java 不允许多继承；就要用对象适配器。就是把 B 作为适配器的成员变量（对象），调用 A 的时候调用 B 的方法；但是适配器不具备原本 B 的能力。感觉也有点类似代理模式，只不过是代理模式需要在前后进行一些操作，这里的话不需要。

什么时候需要用到适配器模式

  • 想复用现成的类，但又不能直接用；那就套个适配器；
  • 版本升级兼容：比如我想把api全部升级到 2.0，但是还有一些程序在用 1.0 的接口，那我就加个适配器，把原本调用 1.0接口转换到调用 2.0；
  • 适配器模式完美契合了对修改关闭、对扩展开放的设计原则
  • 我愿把适配器模式称为嫁接艺术

缺点：

  • 适配器越来越多，直接完犊子了，还不如重构；
  • 感觉就是适配器模式是一种权宜之计，如果良好的设计应该是不太需要用到适配器模式的；

模板方法模式

就是定义了一套模板，有一些相关的抽象方法和一些具体执行这些抽象方法的流程方法，然后用户去实现抽象方法就行；感觉这个设计模式比较简单和通用，像 JDBCTemplate，spring封装了从加载驱动，创建连接，执行sql，获取结果集，释放连接等，我们只需要去实现具体sql语句就行。

抽象类的作用是确定好处理的流程；抽象方法的执行顺序由抽象类提供方法实现。

感觉模版方法模式都不用怎么讲，有过一点抽象思维的人应该都用过；

为什么要叫模版方法呢，就是因为有个templateMethod方法去调用其他抽象方法，子类去实现调用；说白了就是流程定好了，你在我的流程内爱咋咋滴都行。

这种设计模式感觉用在 SPI（Service Provider Interface）比较多，通用化逻辑处理，子类具体化处理实现；

这里也简单讲一讲 SPI，通常情况下，我们实现一套功能，然后对外提供接口；SPI 则是我定义好一系列的接口规范，怎么调用这些接口，让服务提供者去实现这些接口，像我们常见的数据库驱动/日志就是这么实现的。

工厂方法模式

模版方法模式是规定处理流程，子类实现具体处理逻辑；工方法模式则是用来创建实例，就是定义好创建实例的流程，调用后得到一个实例；

这个时候就有小可爱会问了，我直接调用类的构造方法不就可以得到一个实例了吗，干嘛要这么麻烦；对此，我的理解是：其实你也可以把构造方法当成一种工厂方法模式，因为创建一个实例要经过：（1）类不在我就双亲委派；走一遍类加载过程啦（2）分配内存，什么指针碰撞啦，预分配啦；（3）赋默认值；（4）引用指向，赋值啦（过程记不清了，可以去看《码出高效》）

你看，上面这一大坨步骤都不用你来做，你给几个参数就得到了一个实例，那工厂是谁？JVM！我瞎说的。

另一种理解是，在实际业务中，我门创建实例有时候不仅仅是拿到这个实例，可能还会把该实例注册到其他地方，加入到某个容器之类的操作；那么用工厂方法模式的话，这些事情就可以封装起来交给工厂去做，我们只关注我们需要的实例以及接下来如何使用这个实例，其他事情我们不需要感知。这，就是服务到位，客户要啥给啥，不要让客户多做他不care的事儿，以客户为中心！技术源于生活。

到这里聪明的小伙伴可能会意识到，创建实例也要一系列步骤，那岂不是，那岂不是～没错，工厂方法模式套娃了模版方法模式，因为创建实例的流程也是可以编排的嘛。

再换一种有趣的方法来理解，你要写 10 遍 “接美团sp”，没有模版，我们是不是得徒手写，如果我整点活字印刷术，是不是直接起飞。这就是工厂方法模式。

原型模式

原型模式，就是通过复制/拷贝的方式创建实例。

为什么需要原型模式？

  • 复制/拷贝，我们不需要知道具体的内容，直接clone就好；解耦框架和生成的实例，就是生成实例的框架不依赖具体的类；就是你给我啥，我还你啥，不关心你是啥。
  • 创建实例的参数/过程太复杂，难以通过类来生成。
  • 书里还谈到了一点，对象种类繁多，无法整合到一个类中时，可以用原型模式，但是我还不理解是什么意思。

说实话，原型模式的实现不太复杂，但是还是不太理解原型模式的意义和应用场景，后面再慢慢去悟吧。

  • client维护一个map结构来register相关的实例；然后通过调用create方法，create来自Prototype，ConreatePrototype实现了相应clone方法来复制对象。

这里要注意的是，实例要实现clone方法（该方法在Object，cloneable只是一个标记接口），不然会抛出异常。原型模式的特点在于生成实例不依赖类。

建造者模式

这个是用的比较多的设计模式了。但是我看着看着，它好像又跟我平时的用法不太一样，看来还是理解的不够深刻。

有没有觉得这个图跟模版方法模式有点像。 director 的construct调用 builder 的方法，执行一整套builder的流程，跟模版方法模式的templateMethod执行一整套流程很类似；用哪个builder类来初始化director，他就执行哪个builder的流程。

  • 应用场景跟模版方法模式有什么区别；
  • 我们通常创建实例用的builder跟这个是什么关系；

我的理解，builder的本质应该是把一个复杂的对象，拆分成各个独立的部分，然后将他们组装起来。

桥接模式

将类的功能层次结构与实现层次结构分离。

功能层次结构：就是父类功能不够用了，我整个子类继承和扩展父类的能力；

实现层次结构：就是子类实现父类的功能，增加新的实现

顶级接口包含实现；refinedAbstraction是功能的扩展；新增功能增加refinedAbstraction即可；Implementor是抽象类，你可以看到设计模式中大量使用了依赖抽象而不是依赖实现。

所以，当我们使用基本功能，这是通过ConcreteImplementor来初始化Abstraction；需要用到扩展功能，则是用ConcreteImplementor来初始化RefinedAbstraction；

疑问：

  • 为什么Abstraction和Implementor是一种委托关系；而不是继承关系？委托是弱关联，只有在真正实例化Abstraction时，两者才建立联系。
  • 桥接模式的作用是啥？前面提到的分离，到底是为了啥？我的理解是这样的，功能和实现分离，我想增加新的实现，那我就增加ConcreteImplementor，在初始化Abstraction时替换impl就好；我想增加新的功能；那我就增加refinedAbstraction，直接调用就好，impl还是按需传入了；就是扩展不需要修改。
  • 那么如果我们把这一整块耦合在一起，会怎么样？当我增加一个新功能，在Implementor加个默认实现，还是在某个ConcreteImplementor？当我增加新的子类，应该从哪里出发？我感觉这个需要在我们的实际开发中很常见，桥接模式应该好好掌握一下子。

策略模式

策略模式应该是比较简单的设计模式了。1 （）1 = ？根据不同场景（）传入具体的计算策略。发现《图解设计模式》这本书喜欢举一些复杂的例子，讲一大堆跟设计模式无关的话。

  • Context就是上下文，就是使用strategy的类；contextMethod里面会用到strategy；两者之间还是委托关系；感觉策略模式比较好懂，不需要过多解释。

组合模式

组合模式是把一组对象当成一个对象，有点类似套娃，比如 1+ （1+ 2），我们可以把（1+2）看成一个操作数；亦或是文件夹下还有文件夹/文件；这就是组合模式；所以它又叫部分整体模式；一种递归/树形结构；

  • 以上，Leaf可以理解为文件，Composite可以理解为文件夹；两者都是继承Component；要注意对于add/getChild等方法，Leaf是不支持，所以通常在父类我们是以抛出异常，而不是抽象方法来实现。
  • 要注意到，在使用组合模式时，其Leaf和Composite的声明都是实现类，而不是接口，违反了依赖倒置原则。并且在实际应用时，我们不会使用到Component，而是直接使用实现类，Component的作用是保持一致性，比如在Composite的add方法传入的Component类型；是我肤浅的理解，错了当我没说。
  • 组合模式的递归结构真的很帅，以前没有好好掌握真是可惜了。

六大设计原则

  • 单一原则
  • 接口隔离原则：我感觉这个就是单一原则的补充，反正你一个接口就实现一个功能，别揉在一起；
  • 依赖倒置原则：不依赖实现，依赖抽象
  • 开闭原则：修改关闭，扩展开放
  • 里氏代换原则：就是子类必须具备父类的能力，才能去继承父类；否则两者不应该是继承关系；
  • 迪米特原则：最少知识原则，类与类的交换，不需要知道的，那就不用知道；没必要直接交互的，那就通过第三方；
