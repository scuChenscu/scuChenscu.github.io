H厂实习期间需求开发总结

设计模式 多线程
技术 研发

前言

H厂实习期间负责的需求开发工作：在开发过程中用到了一些设计模式和线程池技术。

项目背景

智能检测系统：用户侧的网络设备出现了故障，在系统提交工单：上游系统根据工单获取用户网络设备数据；我负责的数据处理系统负责处理用户数据；下游系统负责基于数据推理网络设备故障根因。

管道模式

背景

  1. 需求分析：用户数据处理包含数据校验、数据筛选、数据转换、数据传输等等一系列步骤。步骤之间存在先后顺序和数据依赖。
  2. 协同开发：同事负责数据处理，我负责搭建整个系统框架；原因：数据处理依赖一定的专家知识和经验。

管道模式简介

管道模式（Pipeline Pattern） 是责任链模式（Chain of Responsibility Pattern）的常用变体之一。在管道模式中，管道扮演着流水线的角色，将数据传递到一个加工处理序列中，数据在每个步骤中被加工处理后，传递到下一个步骤进行加工处理，直到全部步骤处理完毕。

管道模式实现

具体的管道模式实现主要包括三个核心内容：

  1. 定义任务上下文，任务上下文用于串联子步骤之间的数据交互；
  2. 定义处理器接口，每个子步骤都实现该接口中的处理方法：方法内部实现具体的数据处理逻辑；
  3. 配置路由表：路由表指定子步骤的执行顺序；
  4. 定义回调方法，根据数据处理结果进行相应处理：失败则只写库，成功则调用下游接口；

当上游系统发送用户数据，启动推理任务时：创建任务上下文，将任务数据写入任务上下文，根据路由表执行子步骤的数据处理逻辑。

定义任务上下文

任务上下文结合具体的业务逻辑去定义，它包含整个数据处理过程中所需要用到的数据项，同时也可以包含一些重要的日志信息，如任务耗时、异常信息等；

定义处理器接口

接口有个handle()方法，入参是TaskContext，子处理器通过实现该方法来进行相关的业务逻辑处理；返回值为boolean类型，用于表明该子步骤执行成功/失败；

public interface ContextHandler {
    /**
     * 处理任务上下文
     *
     * @param context 上下文
     */
    boolean handle(TaskContext context);
}

配置路由表

通过配置类来指定子处理器的执行顺序

@Configuration
public class HandlersConfig implements ApplicationContextAware {
    private ApplicationContext applicationContext;

    @Override
    public void setApplicationContext(@NotNull ApplicationContext applicationContext) throws BeansException {
        this.applicationContext = applicationContext;
    }

    @Bean("contextHandlers")
    public List<? extends ContextHandler> getHandlers() {
    	// 指定子处理器的调用顺序
        return Stream.of(SubContextHandlerA.class, SubContextHandlerB.class, SubContextHandlerC.class)
            .map(applicationContext::getBean)
            .collect(Collectors.toList());
    }
}

服务调用
@Service
@Slf4j
public class AsyncService {
	// 注入处理器对象列表
    @Resource
    private List<? extends ContextHandler> contextHandlers;

    /**
     * 启动异步任务
     *
     * @param taskContext 任务上下文
     */
    @Async(ExecutorConfig.ASYNC_TASK) // 异步任务
    protected boolean runTask(TaskContext taskContext) {
        // 标志任务管道是否畅通
        boolean isSmooth = true;
        for (ContextHandler handler : contextHandlers) {
            try {
                isSmooth = handler.handle(taskContext);
            } catch (Exception ex) {
                log.error("runTaskError", ex);
                isSmooth = false;
            }
            if (!isSmooth) {
                break;
            }
        }
        return isSmooth;
    }
}
结果处理
// callback是一个consumer
public class Callback implements BiConsumer<TaskContext, Boolean> {

    @Override

        log.info("Callback, taskId = [{}]", context.getTaskId());
        if (isSmooth) {
        	log.info("run task {} success", context.getTaskId());
          // 成功处理
          return;
        }
        // 异常处理
    }
}

private final Callback callback;
// 获取子步骤执行结果
boolean isSmooth = runTask(taskContext);
// 对结果进行处理
callback.accept(taskContext, isSmooth);

设计模式最佳套路2 —— 愉快地使用管道模式

问题

  1. 管道模式和责任链模式的区别

    责任链模式在链上只会有一个处理器用于处理数据：数据到达处理器，处理器无法处理数据，继续调用后续的处理器，如果可以处理数据，则不再往后提交；而管道模式多个处理器都会处理数据。

  2. 哪些中间件用到了责任链模式

    HTTP请求中间件；Spring Security

  3. 管道模式和把每个子步骤抽取成方法的区别

    （1）思想上，一个是面向对象的思想，一个是面向过程的思想

    （2）两者相比较的话，方法实际上就是一个子步骤，但是对于一些异常处理、打印日志和获取数据状态等，用管道模式的话，可以结合代理模式（面向切面编程）的思想在handle()方法前后进行切入处理，实现得更加优雅。

  4. 管道模式的优点

    （1）高内聚低耦合，子步骤间仅通过context进行数据交互；方便支持多人并行开发；

    （2）子步骤顺序方便调整和增删，扩展性强；

    （3）模块重用：如有需要，可以直接对子步骤模块进行重用；

    （4）便于单测：每个子步骤单测十分方便；

    （5）定位异常：子步骤出现异常可以快速抛出，且便于定位；

  5. 管道模式的缺点

    （1）context的维护成本，随着版本迭代，很有可能导致context承载的内容越来越臃肿，不利于维护；因为context不能抽象成领域模型，违背面向对象的思想。

    （2）每个任务都需要经过所有的子步骤，未能做到最优性能；

    （3）声明式编程，对整体的数据流不太好掌握（因为容易陷入局部，可能某些数据会牵一发而动全身）；

  6. Resource注解和Autowired注解的区别

    @Bean("contextHandlers")
public List<? extends ContextHandler> getHandlers() {
  // 这里指定子处理器的调用顺序
  return Stream.of(SubContextHandlerA.class, SubContextHandlerB.class, SubContextHandlerC.class)
    .map(applicationContext::getBean)
    .collect(Collectors.toList());
}    

@Resource // 如果用@Autowired得到的contextHandlers不是上面定义的@Bean("contextHandlers")
private List<? extends ContextHandler> contextHandlers;

    核心区别：Resource默认byName自动装配，Autowired默认按byType自动装配，不支持byName装配，所以上面的contextHandlers如果使用Autowired注入，它得到的顺序不会是我们定义的@Bean(“contextHandlers”)，Autowired会找到所有的ContextHandler写入List（debug的时候看到好像是按字典序）；如果需要指定顺序，则需要在每个ContextHandler实现类使用Order注解。

线程池技术

背景

  1. 下游接口平均耗时3s，难以支持同步调用；（这个应当不能作为使用线程池技术的原因，因为和下游的交互使用的是异步的方式，不存在阻塞）
  2. 系统要求并发量达到100QPS；任务耗时1-2s左右：根据不同的数据量耗时不同；

线程池实现

  1. 继承SpringBoot封装的ThreadPoolTaskExecutor实现自定义线程池（监控日志、线程工厂、拒绝策略）；
  2. 配置：（1）线程池核心参数配置，压测是否满足性能指标；（2）异步配置；项目入口增加@EnableAsync注解，方法入口增加@Aysnc注解；
自定义线程池

自定义线程池主要包含以下几个步骤：

  1. 继承SpringBoot封装的ThreadPoolTaskExecutor类，按需实现自己的线程池（通常是对某些方法进行重写）；
  2. 定义线程池配置类，该类由SpringBoot容器进行管理，主要用于配置线程池核心参数和创建使用线程池对象；
继承SpringBoot封装的ThreadPoolTaskExecutor类

ThreadPoolTaskExecutor的底层实现是Java的ThreadPoolExecutor，执行任务使用submit()方法。线程池状态监控有以下3种实现方式：

  1. 定义showThreadPoolInfo()方法；重写beforeExecute()方法；方法内部调用showThreadPoolInfo()方法；（这个是我后面看线程池源码才注意到的方法，本质上类似代理模式的实现）

  2. 定义showThreadPoolInfo()方法；重写submit()方法，方法内部调用showThreadPoolInfo()方法：

@Slf4j
public class MyExecutor extends ThreadPoolTaskExecutor { // 继承ThreadPoolTaskExecutor 

    public void showThreadPoolInfo() { // 打印线程池的状态日志
        ThreadPoolExecutor threadPoolExecutor = getThreadPoolExecutor();
        log.info("taskCount [{}], completedTaskCount [{}], activeCount [{}], queueSize [{}]",
            threadPoolExecutor.getTaskCount(), threadPoolExecutor.getCompletedTaskCount(),
            threadPoolExecutor.getActiveCount(), threadPoolExecutor.getQueue().size());
    }

    @Override
    public <T> Future<T> submit(Callable<T> task) {
      	showThreadPoolInfo();
        return super.submit(task);
    }
}
  1. 定义showThreadPoolInfo()方法，使用面向切面编程的方式，调用submit()方法前调用showThreadPoolInfo()方法；这种实现无代码侵入：
@Aspect // 切面类
@Component
public class ExecutorAspect {
	// 切点：在线程池执行submit()方法前打印线程池的状态
    @Before("execution(* com.janson.rootcauseinferenceservice.executor.MyExecutor.submit(..))")
    public void showThreadPoolInfo() {
        MyExecutor myExecutor =
            (MyExecutor) RootCauseInferenceServiceApplication.applicationContext.getBean("threadPoolTaskExecutor"); // 获取到线程池的Bean对象
        myExecutor.showThreadPoolInfo();
    }
}
定义线程池配置类

对于核心参数的配置需要基于具体的业务场景：主要考虑的因素：（1）性能要求：QPS100（2）机器资源（3）任务类型；我的机器资源比较充足：单机8核16G；任务类型主要是以CPU为主，少量IO，整体耗时1s左右；我设置核心线程数为8 * 1.5 = 12；因为内存资源充足，直接把阻塞队列设置为100；最大线程数设置为20，比核心线程稍大即可。

@Configuration
public class ExecutorConfig {

    // 核心线程池大小
    public static int corePoolSize = 12;

    // 最大可创建的线程数
    @Value("${async.executor.thread.max_pool_size}")
    private int maxPoolSize = 20;

    // 队列最大长度
    @Value("${async.executor.thread.queue_capacity}")
    private int queueCapacity = 100;

    // 线程池维护线程所允许的空闲时间
    @Value("${async.executor.thread.keep_alive_seconds}")
    private int keepAliveSeconds = 300;

    // 线程池名前缀
    @Value("${async.executor.thread.threadNamePrefix}")
    private static final String threadNamePrefix = "Async-Service-";
  
  	private static final String ASYNC_TASK = "AsyncTask";

    @Bean(name = ASYNC_TASK)
    public ThreadPoolTaskExecutor threadPoolTaskExecutor() {
        ThreadPoolTaskExecutor executor = new MyExecutor(); // 继承ThreadPoolTaskExecutor自定义的线程池 
        executor.setMaxPoolSize(maxPoolSize);
        executor.setCorePoolSize(corePoolSize);
        executor.setQueueCapacity(queueCapacity);
        executor.setKeepAliveSeconds(keepAliveSeconds);
        executor.setThreadNamePrefix(threadNamePrefix);
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.AbortPolicy()); // 拒绝策略
        // 初始化
        executor.initialize();
        return executor;
    }
}

springboot线程池的使用和扩展

问题

  1. 什么情况下使用多线程？如果是串行任务，可以使用多线程吗？

    （1）任务可以拆分成多个水平的小任务，每个小任务并行处理：forkjoinpool；

    （2）IO操作，避免阻塞，使用线程去执行IO操作；

    （3）对于CPU密集型的串行任务，单任务使用多线程没有意义；反而可能因为上下文切换降低性能；

    （4）我的任务场景使用多线程是因为上游任务到达，本地创建任务后直接返回，这里就需要异步去实现了；

  2. 为什么使用线程池而不是SpringBoot的事件监听机制？

​ 两者本质上是一样的，SpringBoot的事件监听机制实际上也是使用线程池去执行。

泛型和模板模式

背景

下游推理系统无法保证推理耗时，为了保证开发工作能够顺利推进，经过对齐之后：同时提供同步接口和异步接口；

对于同步接口，上游系统发起推理请求，本地系统数据处理完成调用下游接口，下游接口推理完成直接返回，本地系统再向上返回结果；

对于异步接口，上游系统发起推理请求，本地系统成功创建推理任务即返回，再异步去调用下游接口，下游接口成功创建推理任务即返回；本地系统为下游提供一个回调接口和为上游提供一个查询接口；

实现

基于模板方法模式来同时实现同步和异步接口，最大程度减少代码冗余，同时保证某一个接口需要下线时最小程度影响另外一个接口（即不需要修改代码）；

问题

  1. 在异步接口实现中，为什么回调需要本地系统参与，直接下游系统和上游系统交互不就可以了吗？

    这个主要基于设计的角度来考虑，上游系统是跟业务强相关的，而下游推理系统是业务不相关的，所以在设计阶段就引入了数据处理系统来对上下游强解耦；不允许两者直接交互。

任务丢失

背景

目前该服务作为试点服务，是单机部署，如果出现单点故障，会导致正在运行的任务丢失。

实现

为了保障任务不丢失，主要考虑了两种方案：（1）根据任务执行的平均耗时，设定超时时间；数据库记录任务创建时间，使用线程定时去扫描数据库任务状态，当任务为running状态但是执行时间已经超时，则判断任务超时，取出任务重跑；该策略的核心问题在于如何去评估任务的执行时间，如果部分任务确实偏离了平均时间，重跑会导致任务状态混乱；（2）使用本地缓存，创建任务写入数据库成功之后，把任务ID写入本地缓存（先写数据库再写缓存），当服务重启，缓存丢失，使用线程去扫描数据库任务状态，当任务状态为running状态，但是本地缓存不存在该任务ID，则说明该任务出现了异常，取出重跑；

补充

以上的实现都是基于单机场景，如果是分布式场景，则需要（1）使用第三方缓存，如Redis；（2）管理者监控，如zookeeper，所有服务器与zookeeper维持心跳，如果某服务器出现异常，及时将任务转发到其他服务器进行重跑；可根据写入第三方缓存判断异常任务：筛选出服务器ID，running状态的任务。

任务中断

背景

如果某些任务在执行的过程中陷入了死循环，需要提供机制来检测和中断异常任务。

实现

线程池不支持任务超时中断，如果需要，则该用Future，但是我们的业务场景不需要使用Future；所以暂时没有想到比较好的方式。

补充

前面提到，异步任务是IO密集型，要发好几个http请求，所以对于这个异步任务的处理，我们还需要考虑一个很关键的因素：网卡的压力；如果同一时间处理多个任务可以考虑加个随机一点点ms的变量 ，稍微错峰，避免同一时间相互挤占网卡；这里只是提出了一个用技术手段错位qps的小技巧。
