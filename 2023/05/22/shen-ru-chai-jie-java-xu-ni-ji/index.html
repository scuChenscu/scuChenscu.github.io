Java虚拟机基本原理

Java代码是怎么运行的

为什么Java要在虚拟机里运行

因为Java是高级语言，它具有跨平台的特性，先经过编译器转换成字节码（字节码具有跨平台的特性），再将字节码转换为机器码，由CPU执行

Java虚拟机具体是怎样运行Java字节码的？

分为两个角度，一个是从虚拟机的角度，一个是硬件的角度

  • 从虚拟机的角度，JVM执行的字节码；.java文件编译后得到.class文件，将.class文件加载到JVM的方法区里面（方法区是JVM规范，目前主流的实现是元空间），然后JVM执行方法区中的代码。
  • 从硬件的角度，字节码是无法被机器直接执行，因为CPU只认识机器码；所以JVM需要把字节码转换为机器码；转换主要有两种方式，一种是解释执行，就是逐条字节码翻译成机器码执行；一种是即时编译（Just-In-Time complilation，JIT），是将一个方法的字节码编译成机器码后执行（之前一直理解成整个.class编译，没想到是以方法为单位进行编译）。

    解释执行的话，优势就是不用等待编译，直接就执行，缺点就是每次都要解释；编译执行的话，就是一开始会比较慢，因为要把整个方法的字节码编译成字节码，但是整个编译完之后，实际运行速度更快。
  • 编译与解释混合执行；解释器启动时先解释执行，省去编译时间；JVM分析热点代码，将高频模块转换为机器码执行。

Java虚拟机时如何加载Java类的

加载

通过双亲委派机制来加载类，即.class文件加载到内存中变成字节流。（类的唯一性是由类加载器实例以及类的全名一同确定的。）

链接

链接分为三个阶段

  • 验证，确保被加载的类符合JVM的约束条件
  • 准备，（1）为类的静态字段分配内存（2）构造其他跟类层次相关的数据结构，比如说用来实现虚方法的动态绑定的方法表。在class文件被加载至Java虚拟机之前，这个类无法知道其他类及其方法、字段所对应的具体地址，甚至不知道自己方法、字段的地址。因此，每当需要引用这些成员时，Java编译器会生成一个符号引用。在运行阶段，这个符号引用一般都能够无歧义地定位到具体目标上。举例来说，对于一个方法调用，编译器会生成一个包含目标方法所在类的名字、目标方法的名字、接收参数类型以及返回值类型的符号引用，来指代所要调用的方法。
  • 解析，符号引用解析成实际引用；碰到未加载的类。如果符号引用指向一个未被加载的类，或者未被加载类的字段或方法，那么解析将触发这个类的加载（但未必触发这个类的链接以及初始化）
  • 初始化，静态字段赋值，如果是final的话，由JVM完成初始化；其他赋值或者静态代码块，执行\方法。

JVM是如何执行方法调用的

主要涉及到的核心概念就是静态绑定和动态绑定

  • Java虚拟机中的静态绑定指的是在解析时便能够直接识别目标方法的情况
  • 动态绑定则指的是需要在运行过程中根据调用者的动态类型来识别目标方法的情况（重写）

方法调用指令共有5种

  1. invokestatic：用于调用静态方法。
  2. invokespecial：用于调用私有实例方法（就是私有非静态方法）、构造器，以及使用super关键字调用父类的实例方法或构造器，和所实现接口的默认方法：\方法；私有方法；super.method()
  3. invokevirtual：用于调用非私有实例方法。
  4. invokeinterface：用于调用接口方法。
  5. invokedynamic：用于调用动态方法。（单独再说）

对于1.和2.的方法调用，属于静态绑定；在类加载的解析阶段，会将符号引用转换为指向方法内存地址的实际引用；
对于3.和4.的方法调用，属于动态绑定；在类加载的解析阶段，会将符号引用转换为指向方法表的对应方法的引用；


实际上，使用了方法表的动态绑定与静态绑定相比，仅仅多出几个内存解引用操作：访问栈上的调用者，读取调用者的动态类型，读取该类型的方法表，读取方法表中某个索引值所对应的目标方法。

Java虚拟机是怎么实现synchronized的

首先，字节码层面的实现：

  • 如果是synchronized(object)，则得到的字节码会有显式的monitorenter指令和monitorexit指令；
  • 如果是作用于方法，字节码中方法的访问标记包括ACC_SYNCHRONIZED。该标记表示在进入该方法时，Java虚拟机需要进行monitorenter操作。而在退出该方法时，Java虚拟机均需要进行monitorexit操作。

monitorenter指令和monitorexit指令都是JVM的指令；抽象理解的话，就是每个锁对象，比如synchronized(object)中的object，它会有一个锁计数器和一个指向持有该锁的线程的指针；（为什么叫monitor，因为每一个对象都有一个monitor与之关联）如果目标锁对象的计数器为0，那么说明它没有被其他线程所持有。在这个情况下，Java虚拟机会将该锁对象的持有线程设置为当前线程，并且将其计数器加1。在目标锁对象的计数器不为0的情况下，如果锁对象的持有线程是当前线程，那么Java虚拟机可以将其计数器加1，否则需要等待，直至持有线程释放该锁。当执行monitorexit时，Java虚拟机则需将锁对象的计数器减1。当计数器减为0时，那便代表该锁已经被释放掉了。（可重入锁）

从上面的理解，我们很直接地想到，当一个锁已经被其他的线程持有时，当前线程执行monitorenter需要等待；那么这个等待过程，在重量级锁的实现下，就是阻塞挂起当前线程，待持有锁的线程释放了锁，当前线程再被唤醒加入到锁的争抢，即执行monitorenter指令。而在这个过程（阻塞和唤醒），会涉及到操作系统从用户态切换至内核态。

轻量级锁

当进行加锁操作时，Java虚拟机会判断是否已经是重量级锁。如果不是，它会在当前线程的当前栈桢中划出一块空间，作为该锁的锁记录，并且将锁对象的标记字段复制到该锁记录中。然后，判断锁标记是否为01（无锁/偏向锁），如果是，Java虚拟机会尝试用CAS（compare-and-swap）操作替换锁对象的标记字段为栈中锁记录的地址，成功则获得锁（我终于理解这个步骤了，轻量级锁的00是因为锁记录的地址内存对齐后，这两位为00，也就是说CAS修改完锁记录的地址，锁标记也一并修改为00）。
如果锁标记不是01，即为00；有两种可能：（1）锁重入，将栈中的锁记录清零；这样在释放锁的时候知道是重入（2）其他线程持有锁，升级锁为重量级锁，然后当前线程自适应自旋尝试获得锁，失败后入队，等待被唤醒；Synchronized 轻量级锁会自旋？好像并不是这样的。

轻量级锁解锁

当进行解锁操作时，如果当前锁记录的值为0，则代表重复进入同一把锁，直接返回即可。否则，Java虚拟机会尝试用CAS操作，比较锁对象的标记字段的值是否为当前锁记录的地址。如果是，则替换为锁记录中的值，也就是锁对象原本的标记字段。此时，该线程已经成功释放这把锁。如果不是，则意味着这把锁已经被膨胀为重量级锁。此时，Java虚拟机会进入重量级锁的释放过程，唤醒因竞争该锁而被阻塞了的线程。

偏向锁

在最初状态下，对象的偏向锁标记为0，表示无锁状态（假设是支持偏向锁）；此时线程执行monitorenter命令，通过CAS操作将当前线程的地址记录在锁对象的标记字段之中，并且将标记字段的最后三位设置为101（进入偏向锁阶段）。
在接下来的运行过程中，每当有线程请求这把锁，Java虚拟机只需判断锁对象标记字段中：（1）最后三位是否为101，（2）是否包含当前线程的地址，（3）以及epoch值是否和锁对象的类的epoch值相同。如果都满足，那么当前线程持有该偏向锁，可以直接返回。如果不满足，判断锁对象的epoch值是否和类的epoch相等，如若不等，那么当前线程可以将该锁重偏向至自己；否则撤销偏向锁，升级为轻量级锁

偏向锁加锁

栈帧创建Lock Record，Lock Record的obj字段指向锁对象；CAS写入ThreadID

偏向锁重入

栈帧创建Lock Record，Lock Record的obj字段指向锁对象；判断ThreadID为当前线程ID，结束；

偏向锁解锁

偏向锁解锁不会主动释放锁，即不会主动修改Mark Word，只会移除栈帧中的Lock Record；

偏向锁升级为轻量级锁

栈帧创建Lock Record，Lock Record的obj字段指向锁对象；（1）ThreadID不是当前线程ID；（2）锁对象epoch等于类的epoch；（3）持有偏向锁的线程不存活，JVM会维持一个Thread List；持有偏向锁的线程不在同步代码块，即Lock Record为空；满足以上三个条件，撤销偏向锁，升级为轻量级锁：将Mark Word修改为指向当前Lock Record的指针，

epoch是什么

其本质是一个版本号，代表了偏向锁的有效性，epoch存储在可偏向对象的MarkWord中。除了对象中的epoch，对象所属的类class信息中，也会保存一个epoch值。

  • 类的对象锁撤销达到20次（默认值），代版本偏向锁失效，epoch + 1；表示之前那一代的偏向锁已经失效。而新设置的偏向锁则需要复制新的epoch值。为了保证当前持有偏向锁并且已加锁的线程不至于因此丢锁，Java虚拟机需要遍历所有线程的Java栈，找出该类已加锁的实例，并且将它们标记字段中的epoch值加1。该操作需要所有线程处于安全点状态。
  • 类的对象锁撤销达到40次，那么Java虚拟机会认为这个类已经不再适合偏向锁。此时，Java虚拟机会撤销该类实例的偏向锁，并且在之后的加锁过程中直接为该类实例设置轻量级锁。

偏向锁撤销

当请求加锁的线程和锁对象标记字段保持的线程地址不匹配时（而且锁对象的epoch值和类的epoch相等，如若不等，那么当前线程可以将该锁重偏向至自己），Java虚拟机需要撤销该偏向锁。这个撤销过程非常麻烦，它要求持有偏向锁的线程到达安全点safepoint，再将偏向锁替换成轻量级锁。（判断持有偏向锁的线程是否还存活，不存活直接撤销；持有偏向锁的线程的lock record为空，说明不在执行同步方法，直接撤销；将偏向线程所有相关Lock Record的Displaced Mark Word设置为null，然后将最高位的Lock Record的Displaced Mark Word 设置为无锁状态，最高位的Lock Record也就是第一次获得锁时的Lock Record（这里的第一次是指重入获取锁时的第一次），然后将对象头指向最高位的Lock Record，这里不需要用CAS指令，因为是在safepoint。 执行完后，就升级成了轻量级锁。原偏向线程的所有Lock Record都已经变成轻量级锁的状态。）

批量重偏向与撤销

  • 批量重偏向是一个线程对大量同类对象同步初始化，另一个线程再对这些对象进程同步操作；这种情况不宜升级为轻量级锁
  • 批量撤销：生产者/消费者队列，不宜用偏向锁，直接升级为轻量级锁

总结

当调用一个锁对象的wait或notify方法时，如当前锁的状态是偏向锁或轻量级锁则会先膨胀成重量级锁。

happen-before

首先，我们要明确，happen-before是用来表述两个操作的内存可见性；专业的说， 操作X happens-before操作Y，那么X的结果对于Y可见。
happen-before好像就是跟我们平时写代码，先写一条指令，再写另外一条指令，先写的指令肯定先于后写的指令，看起来没什么特别之处。

但是在实际执行的时候，如果不存在数据依赖，指令可能会被重排。
