适配器模式和策略模式

背景

广告投放反馈链路：即商家广告投放到抖音、快手等第三方平台，会有一些日志数据产生，如：曝光、点击等；第三方平台通过以https的形式将日志数据回传到我们的系统，我们的系统需要在入口处将外部数据转化为内部统一的数据格式。

广告平台会提供接入方式，在上面填写url，我们的广告数据就会通过url回传给我们。

实现

结合适配器模式和策略模式来实现数据转化：（1）适配器模式：通过适配器把本不兼容的功能适配到原接口；（2）策略模式：根据不同的场景使用不同的策略；而在这里就是根据不同的数据使用不同的策略转化为统一的数据格式。

定义一个适配器抽象类，用于转换数据
public abstract class Adapter<T extends BaseData> { // 这里其实不需要extends BaseData，这样的实现方式是为了要T限制在数据处理，而不是随便一个入参都行
    public abstract TargetData transform(T t);
}
每个数据源实现适配器
@Component
public class DouyinAdapter extends Adapter<DouyinData> { // 这里泛型指定为对应的数据类型
    @Override
    public TargetData transform(DouyinData douyinData) {
        // 相关数据转换操作
        return targetData;
    }
}
服务调用
@RestController
public class DataController {
    @Autowired
    private DataService dataService;

    @Autowired
    private DouyinAdapter douyinAdapter; // 这里需要传入多个适配器，感觉不太合理

    @PostMapping(value = "/douyin")
    public void handleData(@RequestBody DouyinData data) {
        TargetData targetData = douyinAdapter.transform(data); // 注意，从这个角度看很像是策略模式，但是策略模式的入参是一致的，而我这里要对不同类型的数据进行处理，不可能直接用父类BaseData，所以严格来看，我使用的是适配器模式和策略模式思想的组合
        dataService.handleMyData(targetData);
    }
}

为什么要数据转换：

（1）第三方平台的数据格式不统一，如量纲、枚举映射、类型层级；

（2）早期代码仅针对某个平台来实现，需要适配；

其实前面的适配器模式，改成用转换器模式会不会更好？因为就是一个纯粹的数据转换；

如果我摁要把它解释成适配器模式，应该怎么解释呢？

https://juejin.cn/post/6964570540770066469

我们可以这么吹：

本来我们的 service 接收的是AClass类型参数（就是针对最初接入平台所实现的类型），在service层中调用AClass对象的handle方法；

interface AInterface {
  void handle();
}

class AClass implements AInterface {
  public void handle(){};
}

class Service {
  public handle(AInterface aInterface) {
    aInterface.handle();
  }
}

然后我们现在要接入新的平台，比如说是 BClass；参数不同，无法直接送入到service，那么我们可以怎么做呢？

interface BInterface {
  void handle();
}

class BClass implements BInterface {
  public void handle(){};
}

// service该如何去调用呢？
// 使用对象适配器
class Adapter implements AInterface {
  private BInterface bInterface;
  
  public void handle() {
    bInterface.handle();
  }
}


// 在controller层
Adapter adapter = new Adapter(bClass);
service.handle(adapter);


// 那么策略模式在哪里提现呢？
// 你多个平台数据就有多个适配器；根据你传入的class不同，使用不同的适配器来适配

消息队列

背景

当外部数据转化为内部数据之后，需要基于数据信息对商家进行计费操作，而该计费操作是需要调用其他服务，存在一个网络IO的场景，为了避免在高峰期出现IO问题，所以引入了消息队列：上游数据处理->消息队列->下游计费。这里的消息队列就起到一个削峰的作用。

数量量：QPS6000，每条消息几十到上百字节；

实现

每家公司都会根据具体的业务需求来选型相关的中间件，在此基础上进行二次开发；所以对于这一块的实现，我们实际开发中基本上可以做的就是申请资源，引入相应的SDK就可以了。所以这里更多的是去探讨背后的实现原理，而用的消息队列是Kakfa。

生产端消息不丢

当外部数据进入系统转换为消息之后，把消息发送到Kafka，在这里要做到消息不丢主要是保证消息能够成功发送到Kafka。

下面是生产端消息的情况以及解决方案：

（1）生产端在发送消息前宕机；对于这种情况比较好处理，因为此时还没有形成一条真正的消息；我们只需要给外部响应500错误码就好；同时也要有一些相应的报警和埋点日志产生；如果说是异步调用，那么我们就需要额外实现消息入库的操作保证消息不丢失了，或者说使用日志记录，在系统恢复之后实现补偿机制。

（2）网络波动导致发送失败；M厂使用了集群的方式来解决网络问题导致的某节点不可用问题；消息多次重试发送一个集群失败，则会发送第二个集群：在我们申请资源的时候默认配置了双集群，每个集群4台机器分布在2个机房。两个集群能够做到99.999%的可用性保证。

美团消息队列Mafka

Kafka消息不丢

（1）消息到达Kafka后未落盘，Kafka出现了宕机；所以对于涉及到金额的消息，我们需要在多个节点备份才算做一条信息发送成功；通过request.required.acks = -1配置要求主分区同步到其他所有副本才算是一条消息成功发送。

这里额外说一下，对于Kafka响应生产端的ack丢失的话，生产端会重新发送消息，所以我们在生产端保证的是at least once；下图展示的是Mafka集群和生产端的交互，Mafka是M厂基于Kafka进行二次开发的消息中间件。

消费端消息不丢

消费端的核心问题在于消息不重复消费，对于消息不丢很容易解决，就是消息在消息成功消费之后再提交offset。

消费端消息不重

前面也提到消息是at least once，所以我们在消费端需要进行相应的消费幂等。

（1）利用其他中间件的幂等能力，比如redis的set和mysql的主键约束：消费消息的时候先把消息set到redis再去消费，但是这里又有可能引入redis宕机消息丢失的问题；此外可以基于消息的id或者key来作为数据库的主键，主键约束保证消息不重。

（2）事务，查询和消费消息在同一个事务之内，比如用mysql的当前读，select for update，查无记录再去消费，但是这样可能会导致消费效率降低，同时这里要注意建立相应的索引。

消息积压

我们先来分析一下什么情况下会出现消息积压：

（1）生产能力突然飙升：增加生产者或者数据量突然变大；

（2）消费能力下降：比如下游网络IO出现了问题，消费者宕机等；

对于上面两种情况，如果资源充足，则直接对消息队列进行扩容；还可以将消息取出放入额外的临时队列，不进行业务逻辑处理。对于（1）我们可以增加同等的消费者；对于（2）我们可以对下游服务降级或者熔断来提起消费端能力，但是要注意后续的补偿机制实现。
