MySQL数据结构及底层原理

文中有很多内容参考林晓斌 《MySQL实战45讲》，关于林晓斌的故事，大家可以去八卦一下。

系统架构

大体来说，MySQL可以分为Server层和存储引擎层两部分：

  • Server层包括连接器、查询缓存、分析器、优化器、执行器等，涵盖MySQL的大多数核心服务功能；
  • 存储引擎层负责数据的存储和提取。

数据库事务

注意，这里提到的是数据库事务，而不是MySQL事务，或者说Innodb存储引擎实现的事务。

定义

事务就是要保证一组数据库操作，要么全部成功，要么全部失败。

特性

  • 原子性（Actomicity），事务中的操作序列为一个不可分割整体，要么全部执行成功，要么全部不执行。
  • 一致性（Consistency）事务要确保数据库的状态从一个一致的状态转变成另一个一致状态。一致状态的含义是数据库中的数据应满足完整性约束。
  • 隔离性（Isolation）多个事务并发执行时，彼此之间相互独立。
  • 持久性（Durability）事务成功提交，其对数据库的修改将被永久保存。

隔离级别

  • 读未提交（Read Uncommitted）
    一个事务还没提交时，它做的变更就能被别的事务看到。

  • 读已提交（Read Committed）
    一个事务提交之后，它做的变更才会被其他事务看到。

  • 可重复读（Repeatable Read）
    一个事务执行过程中看到的数据，总是跟这个事务在启动时看到的数据是一致的。当然在可重复读隔离级别下，未提交变更对其他事务也是不可见的。

  • 串行化（Serializable）

    顾名思义是对于同一行记录，“写”会加“写锁”，“读”会加“读锁”。当出现读写锁冲突的时候，后访问的事务必须等前一个事务执行完成，才能继续执行。

隔离级别越高，性能越多；因为会引入更多的锁机制，出现读写锁冲突，导致事务阻塞。

并发异常

并发异常是指在多个事务并发执行时可能会出现的异常；这里要注意，我们前面提到的事务的特性是指我们在执行事务的时候应该要满足的特性：一系列操作要么全部成功，要么失败回滚。而事务并发异常，是事务在不同的隔离级别下可能会出现的问题。

  • 回滚丢失/第一类更新丢失：事务B回滚导致事务A的更新丢失

  • 覆盖丢失/第二类更新丢失：事务B的更新覆盖事务A的更新

  • 脏读：事务A更新后回滚，事务B读到事务A的更新

  • 不可重复读，事务B两次读取数据结果不一致（事务A在该过程对数据进行了修改）

  • 幻读

标准SQL事务隔离级别实现原理

在标准SQL事务隔离级别实现下，涉及了大量的读写锁操作。Innodb存储引擎引入了MVCC机制减少锁操作解决读写冲突。

Innodb事务隔离的实现原理

深入理解MySQL中事务隔离级别的实现原理

事务ID

InnoDB里面每个事务有一个唯一的事务ID，叫作transaction id。它是在事务开始的时候向 InnoDB的事务系统申请的，是按申请顺序严格递增的。

每个事务在启动瞬间，会有一个数组，该数组保存当前正在“活跃”的事务：即启动还未提交的事务；数组里面事务ID的最小值记为低水位，当前系统里面已经创建过的事务ID的最大值加1记为高水位。

MVCC机制

MVCC是指多版本并发控制。MVCC是在并发访问数据库时，通过对数据进行多版本控制，避免因写锁而导致读操作的堵塞，从而很好的优化并发堵塞问题。MVCC的实现原理主要依赖于行记录的隐藏字段、undo log和readview。

强调：MVCC机制主要是应用在读已提交和可重复读隔离级别下的快照读操作，快照读即我们平时最常用的select查询语句。

行记录的隐藏字段
  • trx_id：表示最后更新该行记录的事物ID；
  • roll_pointer：回滚指针，指向这条记录的上一个版本；上一个版本在undo log里面；

MVCC机制下，每行数据都是有多个版本的。每次事务更新数据的时候，都会生成一个新的数据版本，并且把transaction id赋值给最新数据版本的事务ID，记为row trx_id。同时，旧的数据版本要保留， 并且在新的数据版本中，能够通过roll_pointer拿到它，实际上以前版本的数据不是真实存在的，因为如果存数据会耗费太多空间，所以undo log记录的是操作，通过执行操作来计算出历史版本数据（可以参考Redis持久化的AOF和RDB）。

总结：数据表中的每一条记录，可能有多个版本，每个版本有自己的row trx_id，当前版本记录可以通过roll_pointer找到上一个版本记录。

undo log

回滚日志，对数据进行insert,delete,update操作生成的方便回滚的日志；

比如说SQL语句是将4改为3，那么回滚日志就是将3改为4，即记录逆操作；

readview

快照读视图，它没有物理结构，作用是在事务执行期间用来定义“我能看到什么数据”。

在MySQL中，实际上每条记录在更新的时候都会同时记录一条回滚操作。记录上的最新值，通过回滚操作，都可以得到前一个状态的值。假设一个值从1被按顺序改成了2、3、4，在回滚日志里面就会有类似下面的记录。 在查询这条记录的时候，不同时刻启动的事务会有不同的read-view。当没有事务再需要用到这些回滚日志时，回滚日志会被删除（如何判断回滚日志不再需要？其实很简单，看事务持久化到哪了就好，后面再展开说）。

好了，讲到这里，大概清楚MVCC的实现原理，就是根据当前的隔离级别和事务ID计算去你可以看到的数据视图；从而避免了读锁的使用，提高了并发效率；因为标准实现上，读数据要加读锁，别的事务就不能写了，而MVCC则是我看我的视图，你改你的数据。

readView生成

读已提交和可重复读的readView生成时机有所不同；

可重复读的readView

前面提到当前事务启动的瞬间，会创建一个数组，这个数组记录了当前正在“活跃”的事务，即先于当前事务创建，但未提交的事务，如下图中的黄色部分的未提交事务集合，数组最小值为低水位，数组最大值+1 为高水位：

那么，对于每一个行记录的row trx_id，有以下几种情况：

  • 小于低水位，即落在绿色部分，可见；因为是在当前事务之前就已经提交了。（也可能是当前事务自己提交的）
  • 大于高水位，即落在红色部分（这里可能不太好理解，红色部分不是未开始事务吗？当前事务启动之后，又有新的事务启动了，那新事务的id就落在了高水位之后，并且它先于当前事务提交了），很明显对当前事务是不可见的。
  • 介于高低水位之间，即落在黄色部分，此时就需要额外地判断了：
    • row trx_id在数组集合中，即当前事务启动时，更新该row的事务还未提交，那就是对当前事务不可见；
    • row trx_id不在数组集合中，即当前事务启动时，更新该row的事务已经提交了，那就是对当前事务可见；

总结，对于可重复读的readView生成，核心就一句话：以我启动的时刻为准，如果一个数据版本是在我启动之前生成的，就认；如果是我启动以后才生成的，我就不认，我必须要找到它的上一个版 本”。当然，如果“上一个版本”也不可见，那就得继续往前找。还有，如果是这个事务自己更新的数据，它自己还是要认的。

读已提交的readView

读已提交生成readView是在每次执行查询的时候，而不是在事务启动的时候，其余原理同可重复读的readView。所以读已提交会出现不可重复读的情况，因为有些事务介于两次查询之间进行了提交，那些row在第二次查询时就是可见的。

MVCC 的优缺点
  • 优点：前面的分析很清晰了，就是不用上读锁，提高了并发效率；
  • 缺点：存储row trx_id和roll_pointer等需要额外的空间；

快照读和当前读

select * from table where id = ?; # 快照读，不加锁，同时也忽略读写锁；
BEGIN; # START TRANSACTION;两者是等价的
# 下面的语句只有在事务中才有效，通常会先开闭自动提交
select * from table where id = ? lock in share mode; # 加共享锁，即别的事务只能读不能写；如果存在写锁，则阻塞至写锁释放，读取最新值
select * from table where id = ? for update; # 加排它锁，即别的事务不能读和写
快照读

快照读就是我们平时执行的普通的select查询语句；读已提交和可重复读会涉及前面提到的readview视图的概念，两者生成读视图的时机不同。

  • 读未提交：直接返回记录上的最新值，没有视图概念；
  • 读已提交：视图是在每个SQL语句开始执行的时候创建的；
  • 可重复读：视图是在事务启动时创建的，整个事务存在期间都用这个视图；
  • 串行化：直接用加锁的方式来避免并行访问。

而关于视图的创建，就用到了隐藏字段和undo log。

当前读

当前读就是读取当前版本最新的数据；select … in share mode; select for update; insert, delete, update这些操作都是当前读，也就是说，当前“读”的读不仅仅是读操作，也包括在修改数据的写操作中包含的“读”操作。

理论上，对于当前读，我们就按照标准的事务隔离级别实现方式进行加锁就可以了。但是，Innodb默认的隔离级别是可重复读，按照标准的锁实现，我们知道会出现幻读问题，所以Innodb引入了由record Lock和gap Lock组成的Next-key lock；避免可重复读下的快照读出现幻读（这里要注意，实际上并没有彻底解决幻读，但是在面试基本上到这一步就差不多了）。

所以，Innodb事务隔离级别实现方式：

两阶段锁协议

可以看到，在lnnoDB事务中，行锁是在需要的时候才加上的，而是要等到事务结束时才释放。这个就是两阶段锁协议。事务update加行锁，commit释放。

Innodb的锁

读未提交和读已提交只有行级锁；主要讨论可重复读下的行锁和间隙锁。

为什么需要间隙锁

间隙锁是Innodb在可重复读隔离级别下解决幻读；快照读用 MVCC，当前读用行锁和间隙锁；

行锁和间隙锁

InnoDB行锁是通过给索引上的索引项加锁来实现的，这种行锁实现特点意味着：只有通过索引条件检索数据，InnoDB才使用行级锁，否则，InnoDB将使用全表的间隙锁，效果等价于表锁。select加锁分析

  • 行锁（Record Lock）：对行记录加锁。
  • 间隙锁（Gap Lock）：产生幻读的原因是，行锁只能锁住行，但是新插入记录这个动作，要更新的是记录之间的“间隙”。因此，为了解决幻读问题，InnoDB只好引入新的锁，也就是间隙锁(Gap Lock)。就是锁定范围空间的数据，假设id有3,4,5，锁定id>3的数据，是指的4，5及后面的数字都会被锁定，因为此时如果不锁定没有的数据，例如当加入了新的数据id=6，就会出现幻读，间隙锁避免了幻读。
  • Next-Key Lock ：Record Lock + Gap Lock，锁定一个范围并且锁定记录本身 。
当前读的实现方式
  • 对主键或唯一索引，如果当前读时，where条件全部精确命中(=或者in)，这种场景本身就不会出现幻读，所以只会加行记录锁。
  • 没有索引的列，当前读操作时，会加全表gap锁，生产环境要注意。
  • 非唯一索引列，如果where条件部分命中(>、<、like等)或者全未命中，则会加附近Gap间隙锁。例如，某表数据如下，非唯一索引2,6,9,9,11,15。如下语句要操作非唯一索引列9的数据，gap锁将会锁定的列是(6,11]，该区间内无法插入数据。
为什么非索引列出加全表gap锁

不加的话会出现幻读，当你查询条件为非索引列等值查询，只锁该行记录，别的事务可以写入新的值等于非索引列的行记录；幻读就出现了，所以只能是锁表；

间隙锁的危害
  • 死锁，间隙锁不会冲突；也就是可能会出现两个事务对同一个范围加上了间隙锁，再执行更新操作导致死锁；
  • 性能，间隙锁会锁住更大的范围，影响并发；

索引

定义

索引是一种可以提高查询效率的数据结构；

数据结构：B+树（横向比较其他数据结构）

  • 哈希；单点检索效率高，但是区间检索、排序（order by）很差，也许还会有大数据量的哈希碰撞问题。
  • 二叉搜索树，会产生单边树问题形成链表（连最基本的自增主键效果都很差），树高过高，浪费空间。
  • AVL，自平衡二叉查找树，任一节点对应的两棵子树的最大高度差为1，相对于二叉查找树效果有所改善，但还是有树高问题，IO操作高。
  • B树，非叶子节点存储索引+数据，可以存储的索引少，page 16KB。（查询不稳定）
  • B+树，平均查询次数低，查询时间稳定（稳定大于一切）。非叶子节点可以存储更多的索引（非叶子节点只存储索引，不存储数据，每次IO获取更多的索引），全表扫描直接到叶子节点就行，叶子节点有链表，支持范围查询。

B+树索引并不能直接找到具体的行，只是找到被查找行所在的页，然后DB通过把整页读入内存，再在内存中查找。（一个节点就是一个page）

B+树在主键索引树存数据，其他索引树存主键，所以会回表（即先拿主键，再通过主键拿数据）；为了避免回表，可以使用覆盖索引（即需要的数据就在索引）；

采用B+树的索引结构优点：

  1. B+树的高度一般为2-4层，所以查找记录时最多只需要2-4次IO，相对二叉平衡树已经大大降低了。
  2. 范围查找时，能通过叶子节点的指针获取数据。例如查找大于等于3的数据，当在叶子节点中查到3时，通过3的尾指针便能获取所有数据，而不需要再像二叉树一样再获取到3的父节点。
  • 叶子节点存放了整张表的所有行数据。
  • 非叶子节点并不存储行数据，是为了能存储更多索引键，从而降低B+树的高度，进而减少IO次数。
  • 聚集索引的存储在物理上并不是连续的，每个数据页在不同的磁盘块，通过一个双向链表来进行连接。（感觉这句话讲得不太对，这个应该是指块与块之间），聚集索引的特点：查询快，修改慢，所以通常建议自增主键，减少插入行为；
  • 每个表可以有多个辅助索引。
  • 通过辅助索引查数据时，先查找辅助索引获得聚集索引的主键，然后通过主键索引来查找完整的行记录。（回表）
  • 通过非主键索引比主键索引查找速度要慢一倍。

在InnoDB中，数据和索引文件是合起来储存的，InnoDB 的存储文件有两个，后缀名分别是 .frm 和 .idb，其中 .frm 是表的定义文件，而 idb 是数据文件。

索引

定义：一种可以提高数据查询效率的数据结构。

聚簇索引：主键索引，叶子节点存整行数据

非聚簇索引：非主键索引，叶子节点存主键，再根据主键去查主键索引拿到行数据（回表）

覆盖索引：即索引K已经满足了查询要求，不需要再回表。

联合索引：多个key按序检索，最左前缀原则，可以达到覆盖索引的效果，但是索引需要更多的存储空间

MySQL联合索引，最左前缀原则，在遇到范围查询的时候，就会停止匹配。

自增主键作为索引的好处：页分裂，数据移动

最左前缀原则：这个最左前缀可以是联合索引的最左N个字段，也可以是字符串索引的最左M个字符。

索引选择

从索引的定义出发：一种可以提高数据查询效率的数据结构

  • 短小精悍（索引需要额外的存储空间）
  • 重复度低（走索引可以快速过滤掉绝大部分不符合要求的数据）
  • 查询频度高（提高查询效率，当然得优先对高频数据建立索引）
  • 最好是单调递增，非单调会导致索引树分裂

count()

count()的语义：count()是一个聚合函数，对于返回的结果集（存储引擎返回），一行行地判断（Server层判断），如果count函数的参数不是NULL，累计值就加1，否则不加。最后返回累计值。

慢SQL

  • 抖动：系统刷脏页，将redo log写入磁盘
  • 锁，并发
  • 没有走索引

binlog（归档日志）和redo log（重做日志）

聚簇索引

每张InnoDB表有且只有一个聚集索引，那它是怎么选择索引的呢？

  • 一般情况，用PRIMARY KEY来作为聚集索引。
  • 如果没有定义PRIMARY KEY，将会用第一个UNIQUE且NOT NULL的列来作为聚集索引。
  • 如果表没有合适的UNIQUE索引，InnoDB会自动生成一个不可见的列名为ROW_ID，索引名为GEN_CLUST_INDEX的聚簇索引，该列是一个6字节的自增数值，随着插入而自增。

很明显，缺少主键的表，InnoDB会内置一列用于聚簇索引来组织数据。而没有建立主键的话就没法通过主键来进行索引，查询的时候都是全表扫描，小数据量没问题，大数据量就会出现性能问题。所以在建表的时候，如果没有逻辑唯一且非空列时，可以添加一个auto_increment的列，方便建立一个聚集索引。

Redo log（保证事务）记录更新操作，文件满了刷盘

如果每次更新操作直接写把记录写到磁盘：

  1. 磁盘找到对应记录
  2. 更新记录

IO成本，查找成本太高。

更新redo log：

  1. 更新记录追加到log文件
  2. 更新内存

InnoDB的redo log是固定大小的，比如可以配置为一组4个文件，每个文件的大小是 1GB，那么这块“粉板”总共就可以记录4GB的操作。从头开始写，写到末尾就又回到开头循环写。

write pos是当前记录的位置，checkpoint是当前要擦除的位置

从checkpoint到write pos是未写入数据表的更新操作；从write pos到checkpoint是空余的部分。如果write pos追赶上checkpoint表面redo日志已满，需要刷盘。

binlog日志，备份和回滚

Redo log(perpare) -> binlog （备份归档）-> redo log(commit)

两阶段提交

两阶段提交的第一阶段 （prepare阶段）：写rodo-log 并将其标记为prepare状态。

紧接着写binlog

两阶段提交的第二阶段（commit阶段）：写bin-log 并将其标记为commit状态。

所以说，两阶段提交的主要用意是：为了保证redolog和binlog数据的安全一致性。只有在这两个日志文件逻辑上高度一致了。你才能放心地使用redolog帮你将数据库中的状态恢复成crash之前的状态，使用binlog实现数据备份、恢复、以及主从复制。而两阶段提交的机制可以保证这两个日志文件的逻辑是高度一致的。没有错误、没有冲突。

在前面的分享binlog的文章中有跟大家提到过一个参数sync_binlog=1。这个参数控制binlog的落盘时机，并且你们公司线上数据库的该参数一定被设置成了1。

Notice！！！ 这个参数为1时，表示当事物第一阶段提交时会将binlog落盘。

为什么要两阶段提交；如果没有两阶段提交，不管是先写redo log还是先写binlog，一旦再先后之间发生crash，就会出现不一致的问题

崩溃恢复

如果在图中时刻 A 的地方，也就是写入 redo log 处于 prepare 阶段之后、写 binlog 之前，发生了崩溃（crash），由于此时 binlog 还没写，redo log 也还没提交，所以崩溃恢复的时候，这个事务会回滚。这时候，binlog 还没写，所以也不会传到备库。

如果 redo log 里面的事务是完整的，也就是已经有了 commit 标识，则直接提交；如果 redo log 里面的事务只有完整的 prepare，则判断对应的事务 binlog 是否存在并完整：XID

  • a. 如果是，则提交事务；
  • b. 否则，回滚事务。

这里，时刻 B 发生 crash 对应的就是 2(a) 的情况，崩溃恢复过程中事务会被提交。

ps: 两阶段提交的最后一个阶段的操作本身是不会失败的，除非是系统或硬件错误，所以也就不再需要回滚（不然就可以无限循环下去了）

先写redo，因为它可以提交或者回滚

数据库恢复到某个目标时间点：该时间点前的全量备份 + 全量备份时间点至目标时间点的binlog日志

索引失效

一张图搞懂MySQL的索引失效

  • 一旦对索引字段做函数操作，MySQL 会放弃使用索引，如计算、函数、（自动or手动）类型转换等操作，会导致索引失效从而全表扫描
  • mysql在使用不等于（!=、<>）的时候无法使用索引会导致全表扫描（除覆盖索引外）
  • 使用or连接
  • 索引类型为varchar，字符串不加单引号索引失效，如where id = 2000（失效，发生了隐式转换），where id = “2000”（走索引）；索引类型为int，加不加引号都走索引

为什么 int 加不加引号都可以使用索引呢?

因为 int 类型的数字只有 2 能转化为’2’，是唯一确定的。所以虽然需要隐式转换，但不影响使用索引（不是很理解）

varchar和int索引

int占4个字节，如果varchar长度小于4，使用varchar，否则使用int，节省空间。

Memory存储引擎
  • 数据存储在内存中，默认使用hash索引
MyISAM存储引擎
  • 强调性能
  • 不支持事务
  • 不支持行级锁和外键
  • 读操作速度很快（也许是因为非聚集索引，不需要回表，所以读性能很高）
  • 非聚集索引，使用 B+ 树作为索引结构，索引和数据文件分离，主键索引和辅助索引独立
MyISAM与InnoDB的区别

存储结构

MyISAM：每个MyISAM在磁盘上存储成三个文件。第一个文件的名字以表的名字开始，扩展名指出文件类型。.frm文件存储表定义。数据文件的扩展名为.MYD (MYData)。索引文件的扩展名是.MYI (MYIndex)。
InnoDB：所有的表都保存在同一个数据文件中（也可能是多个文件，或者是独立的表空间文件），InnoDB表的大小只受限于操作系统文件的大小，一般为2GB。

数据库三范式

  1. 第一范式：确保每列的原子性:如果每列都是不可再分的最小数据单元（也称为最小的原子单元），则满足第一范式（1NF）

  2. 第二范式：满足第一范式，并且表中非主键列不存在对主键的部分依赖。 第二范式要求每个表只描述一件事情。

  3. 第三范式定义是，满足第二范式，并且表中的列不存在对非主键列的传递依赖。除了主键订单编号外，顾客姓名依赖于非主键顾客编号。

分库分表

为什么要分库分表

数据库基本指标：库物理文件大小<100G；表<100；字段<200；单表记录数<500W

  • 分库
    1. 数据表太多，磁盘空间不足，IO次数太多，CPU过载；
    2. 单机服务器数据库连接数量资源有限，产生性能瓶颈
  • 分表
    1. 表数据量太大，索引膨胀，CURD容易超时；
  • 分库分表
    1. 业务需求，比如：有些财务系统，需要按月份和年份汇总，所有用户的资金。每年都有个单独的数据库，每个数据库中，都有12张表，每张表存储一个月的用户资金数据。
如何进行分库分表

  • 分库

    1. 垂直分库

      （1）根据业务逻辑拆分出不同的数据库，部署到不同的服务器，解决磁盘空间、内存、TPS等问题。

    2. 水平分库

      （1）将单库中的数据拆分到不同的数据库， 水平分库分表能够有效的缓解单机和单库的性能瓶颈和压力，突破IO、连接数、硬件资源等的瓶颈。

  • 分表

    1. 垂直分表

      （1）拆分核心字段和扩展字段

      （2）拆分访问频度高的字段和访问频度低的字段

    2. 水平分表

      （1）RANGE

      （2）HASH取模

      （3）地理区域

      （4）时间

分库分表存在的问题
  1. 分布式事务
  2. 联合查询
  3. 跨库/表join
  4. 多库结果集合并（group by，order by）

主从复制

为什么要主从复制
  1. 数据备份、异地容灾
  2. 读写分离、分担压力
  3. 高可用、可扩展
实现原理

  1. master 服务器会将 SQL 记录通过多 dump 线程写入到 binary log 中。
  2. slave 服务器开启一个 io thread 线程向服务器发送请求，向 master 服务器请求 binary log。master 服务器在接收到请求之后，根据偏移量将新的 binary log 发送给 slave 服务器。（解耦合）
  3. slave 服务器收到新的 binary log 之后，写入到自身的 relay log 中，这就是所谓的中继日志。
  4. slave 服务器，单独开启一个 sql thread 读取 relay log 之后，写入到自身数据中。

MySQL优化方案

  1. 索引优化
    • 只要列中含有NULL值，就最好不要在此例设置索引，复合索引如果有NULL值，此列在使用时也不会使用索引
    • 尽量使用短索引，如果可以，应该制定一个前缀长度
    • 对于经常在where子句使用的列， 最好设置索引，这样会加快查找速度
    • 对于有多个列where或者order by子句的，应该建立复合索引
    • 对于like语句， 以%或者“’开头的不会使用索引，以%结尾会使用索引
    • 尽量不要在列上进行运算(函数操作和表达式操作)
    • 尽量不要使用not in和<>操作
  2. SQL语句的优化
    • 捕获低效sql
      1) slow_ query_ log：这个参数设置为ON，可以捕获执行时间超过一定数值的SQL语句。
      2) ong_ query_ time：当SQL语句执行时间超过此数值时，就会被记录到日志中，建议设置为1或者更短。
      3) slow_ query_ log _ file：记录日志的文件名。
      4) log_ queries_ not _ _using_ indexes：这个参数设置为ON，可以捕获到所有未使用索引的SQL语句，尽管这个SQL语句有可能执行得挺快。
    • 慢查询优化的基本步骤
      1)先运行看看是否真的很慢，注意设置SQL_ NO_ CACHE
      2) where条件单表查， 锁定最小返回记录表。这句话的意思是把查询语句的where都应用到表中返回的记录数最小的表开始查起，单表每个字段分别查询，看哪:个字段的区分度最高
      3)explain查看执行计划，是否与1预期-致(从锁定记录较少的表开始查询)
      4)order by limit形式的sq|语句让排序的表优先查
      5)了解业务方使用场景
      6)加索引时参照建索引的几大原则
      7)观察结果，不符合预期继续从1开始分析
    • 优化原则
      1)查询时，能不要*就不用*，尽量写全字段名
      2)大部分情况连接效率远大于子查询
      3)多使用explain和profile分 析查询语句
      4)查看慢查询日志，找出执行时间长的sql语句优化
      5)多表连接时，尽量小表驱动大表，即小表join大表
      6)在千万级分页时使用limit
      7)对于经常使用的查询，可以开启缓存
  3. 数据库表优化
    1)表的字段尽可能用NOT NULL
    2)字段长度固定的表查询会更快
    3)把数据库的大表按时间或一些标志分成小表
    4)将表水平拆分或垂直拆分

char和varchar的区别

MySQL是一行数据共享空间，默认最大65535字节，2^16-1 = 65535；为什么不是65536？好奇怪，好像是有个 1～2 个字节来表示长度，2 个字节可以表示的最大长度就是65535。列数最大是 4096；varchar(n)，在MySQL5.0 之后，n表示字符数；对于不同的编码，每个字符占的字节数不同；char是定长，取值范围 0～255，用空格来填充，取数据需要trim()
